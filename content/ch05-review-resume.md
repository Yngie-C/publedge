# 자소서 검토와 개선: 세 번째 스킬

지금까지 채용공고를 분석하고, 자소서를 생성하는 스킬을 만들었어요. 이제 생성된 자소서를 검토하고 개선하는 세 번째 스킬을 만들 차례예요.

이 챕터에서 만드는 `/review-resume` 스킬은 단순한 교정 도구가 아니에요. 기존 AI 도구에는 없는 기능, 바로 **환각(hallucination) 검출**이 핵심이에요. "AI가 자소서에 없는 경험을 지어냈는지" 자동으로 잡아내는 기능이죠. 왜 이 기능이 필요한지부터 살펴볼게요.

---

## 왜 AI 검토가 필요한가요?

### AI가 쓴 자소서의 한계

AI는 자소서를 잘 써요. 하지만 "잘 쓴다"는 것에는 함정이 있어요.

**뻔한 표현이 넘쳐나요.** "적극적인 자세로 임했습니다", "팀원들과 소통하며 문제를 해결했습니다" — AI가 학습한 수많은 자소서에서 가장 많이 등장하는 표현들이에요. 채용 담당자는 하루에 수백 개의 자소서를 읽어요. 이런 표현은 첫 문장에서 걸러져요.

**경험을 지나치게 미화해요.** "소규모 스터디를 운영했다"는 경험이 "100명 규모의 커뮤니티를 리드했다"로 변질될 수 있어요. 인상적으로 쓰려는 AI의 성향이 과장을 만들어내죠.

그리고 가장 위험한 문제가 있어요.

**없는 경험을 지어내요.** AI는 그럴싸한 자소서를 쓰기 위해 프로필에 없는 내용을 추가하기도 해요. "Python을 활용한 데이터 분석 프로젝트를 진행했습니다"라고 썼는데, 실제 프로필에는 Python 경험이 없는 경우가 생길 수 있어요. 이것이 바로 AI의 **환각(hallucination)** 문제예요.

### 기존 AI 도구에 없는 기능

ChatGPT, Gemini, 다른 자소서 AI 서비스들도 자소서 검토 기능을 제공해요. 문장을 다듬고, 맞춤법을 교정하고, 더 좋은 표현을 제안하죠.

하지만 이 도구들에는 없는 기능이 하나 있어요.

**"이 자소서의 내용이 실제 당신의 경험과 일치하나요?"를 검증하는 기능이 없어요.**

왜냐하면 기존 도구들은 자소서 텍스트만 보거든요. 당신의 실제 경력이 무엇인지 모르니까, 사실 여부를 확인할 방법이 없어요.

우리 파이프라인은 달라요. `data/my-profile.md`라는 **source of truth**가 있기 때문이에요. 이 파일에 기록된 경험과 자소서 내용을 교차 검증하는 것, 이것이 우리 스킬의 핵심 차별점이에요.

### 비유: 까다로운 면접관의 사전 피드백

이 스킬을 사용하는 건 면접 전날 밤, 대기업 채용 담당자 친구에게 자소서를 보여주는 것과 같아요.

> "이 부분, 면접에서 물어볼 것 같은데 실제로 했어? 수치가 너무 애매한데? 이 표현은 모든 지원자가 쓰는 거야."

냉정하고 까다롭지만, 덕분에 면접장에서 당황하는 일을 미리 막을 수 있어요.

## /review-resume 스킬 만들기

### 핵심 먼저 이해하기

스킬 파일을 작성하기 전에, 이 스킬이 하는 일을 세 문장으로 정리해볼게요.

**첫째, 역할 설정이에요.** 클로드에게 "당신은 대기업 채용 담당자입니다"라고 지정해요. 친절한 교정자가 아니라, 자소서의 허점을 찾아내는 까다로운 검토자 역할이에요. AI에게 역할을 구체적으로 줄수록 더 날카로운 피드백이 나온다는 걸 기억하세요.

**둘째, 세 파일 교차 검증이에요.** `$ARGUMENTS`에서 자소서 파일 경로와 JD 분석 JSON 경로를 받고, `data/my-profile.md`도 함께 읽어요. 자소서 한 파일만 보는 게 아니라, JD 요구사항과 실제 프로필을 동시에 참조해서 검토해요. 세 파일을 동시에 보는 것이 이 스킬의 핵심 구조예요.

**셋째, 환각 검출이 핵심이에요.** `data/my-profile.md`에 기록되지 않은 경험이나 기술이 자소서에 등장하면 ⚠️ 환각 경고로 표시해요. 이것이 이 스킬을 특별하게 만드는 기능이에요.

> **NOTE:** 앞선 챕터에서 `/generate-resume` 스킬을 만들 때 `data/my-profile.md`를 source of truth로 설정한 이유가 여기서 드러나요. 생성 단계와 검토 단계가 같은 기준 파일을 공유하기 때문에, 일관된 파이프라인이 만들어지는 거예요.

### 전체 스킬 파일 작성

`.claude/commands/review-resume.md` 파일을 만들어볼게요. 빌드-리빌 패턴으로 각 섹션이 왜 그렇게 작성됐는지 함께 이해하면서 진행할게요.

~~~markdown
```markdown
# review-resume

당신은 10년 경력의 대기업 채용 담당자입니다. 하루에 수백 개의 자소서를 검토하며,
뻔한 표현과 과장된 내용에 즉각 반응합니다. 친절하지만 냉정하게 피드백을 제공하세요.

## 입력 파일

$ARGUMENTS 형식: {자소서 경로} {JD 분석 JSON 경로}

다음 세 파일을 모두 읽어주세요:
1. {자소서 경로} — 검토 대상 자소서
2. {JD 분석 JSON 경로} — 채용공고 분석 결과 (키워드, 역량, 우대사항)
3. data/my-profile.md — 지원자의 실제 경력 및 경험 (source of truth)

## 검토 기준 (각 20점, 총 100점)

### 1. JD 매칭도 (20점)
- JD 분석 JSON의 핵심 키워드가 자소서에 포함됐는가?
- JD에서 요구하는 역량에 직접적으로 대응하는 내용이 있는가?
- 우대사항에 해당하는 경험이 부각됐는가?

### 2. 구체성 (20점)
- 수치가 포함됐는가? (기간, 규모, 성과 수치 등)
- 추상적인 표현 대신 구체적인 행동과 결과를 서술했는가?
- "열심히 했습니다" 수준의 서술은 없는가?

### 3. 차별화 (20점)
- 다른 지원자와 구별되는 고유한 경험이 포함됐는가?
- "적극적으로", "소통하며", "노력했습니다" 같은 뻔한 표현은 없는가?
- 이 지원자만의 관점이나 인사이트가 담겼는가?

### 4. 문장력 (20점)
- 문장이 간결하고 명확한가? (한 문장에 하나의 메시지)
- 맞춤법과 문법 오류가 없는가?
- 문어체와 구어체가 혼재되지 않는가?

### 5. 구조와 분량 (20점)
- 300~500자 범위를 지키고 있는가?
- 도입-본론-마무리의 논리적 흐름이 있는가?
- 면접관이 끝까지 읽고 싶어지는 구조인가?

## 환각 검출 (필수)

data/my-profile.md를 기준으로, 자소서에 포함된 모든 경험, 기술, 수치, 프로젝트를
하나씩 대조 검증하세요.

각 항목을 다음 형식으로 표시하세요:
- ✅ **확인됨**: 프로필에 근거가 있는 내용
- ⚠️ **환각 의심**: 프로필에서 확인되지 않는 내용 (구체적으로 어떤 내용인지 명시)

환각이 발견된 경우, 프로필에 실제로 있는 유사한 경험으로 대체할 것을 제안하세요.

## 출력 형식

### 종합 점수
| 기준 | 점수 | 만점 |
|------|------|------|
| JD 매칭도 | X | 20 |
| 구체성 | X | 20 |
| 차별화 | X | 20 |
| 문장력 | X | 20 |
| 구조와 분량 | X | 20 |
| **총점** | **X** | **100** |

### 환각 검출 결과
[각 항목 ✅/⚠️ 표시]

### 기준별 상세 피드백
[각 기준에 대한 2-3줄 피드백]

### 수정 제안 (3-5개)
우선순위 순으로 구체적인 수정 방향을 제시하세요.
단순히 "더 구체적으로"가 아니라, 실제 수정 예시를 포함하세요.

출력 파일: output/review-{자소서 파일명}
```
~~~

파일을 저장했으면, 각 섹션이 어떤 역할을 하는지 살펴볼게요.

**역할 지정 부분**을 보면 단순히 "AI 도우미"가 아닌 구체적인 페르소나를 설정했어요. "10년 경력", "하루에 수백 개"라는 표현이 클로드에게 얼마나 비판적으로 봐야 하는지 맥락을 줘요. "친절하게 교정해주세요"와 "10년 경력 채용 담당자처럼 냉정하게 검토하세요"는 완전히 다른 결과를 만들어내요.

**세 파일 교차 검증** 구조가 이 스킬의 핵심이에요. 자소서만 보면 교정 도구에 불과하지만, JD 분석 결과와 실제 프로필을 함께 읽으면서 진짜 검토가 시작돼요. 클로드는 파일 세 개를 동시에 읽고 내용을 비교해요 — 이건 사람도 쉽지 않은 작업이에요.

**100점 기준 채점표**를 넣은 이유도 있어요. 단순히 "좋아요/나빠요"가 아니라 항목별 점수로 나오면, 어떤 부분에 집중해서 고쳐야 하는지 명확해져요. 두 번째 검토에서 어떤 항목이 얼마나 올랐는지도 수치로 확인할 수 있고요.

**환각 검출 섹션**에서 "프로필에서 확인되지 않는 내용"이라는 표현에 주목해요. 클로드에게 프로필을 ground truth로 명확히 지정했기 때문에, AI가 자의적으로 판단하는 게 아니라 실제 파일 내용과 대조하는 거예요. "이상한 것 같다"가 아니라 "파일에 없다"는 사실 기반 판단이에요.

**수정 제안**에서 "실제 수정 예시를 포함하세요"라고 명시한 것도 중요해요. "더 구체적으로 써야 해요" 같은 막연한 피드백이 아니라, 실제로 쓸 수 있는 대안 문장을 제시하도록 유도하는 거예요. 클로드한테 막연한 지시를 하면 막연한 결과가 나와요. 구체적인 출력 형식을 지정할수록 실용적인 피드백이 나오는 거예요.

## 스킬 실행 & 피드백 확인

스킬 파일을 만들었으니 실행해볼게요. 터미널에서 프로젝트 폴더로 이동한 뒤, 클로드 코드를 실행해요.

```bash
cd my-resume-pipeline
claude
```

클로드 코드가 실행되면 스킬을 입력해요.

```bash
/review-resume output/resume-techcorp-backend.md data/analysis/techcorp-backend.json
```

클로드가 세 파일을 읽고 분석을 시작해요. 잠시 후 결과가 출력돼요.

### 점수표 읽기

출력된 결과에서 먼저 점수표를 확인해요.

```markdown
### 종합 점수
| 기준 | 점수 | 만점 |
|------|------|------|
| JD 매칭도 | 16 | 20 |
| 구체성 | 12 | 20 |
| 차별화 | 11 | 20 |
| 문장력 | 17 | 20 |
| 구조와 분량 | 14 | 20 |
| **총점** | **70** | **100** |
```

70점이 나왔어요. 나쁘지 않지만 개선 여지가 있어요. 특히 **구체성(12점)**과 **차별화(11점)**가 낮게 나왔네요. 수치와 고유한 경험이 부족하다는 신호예요.

### 환각 검출 결과 확인

이 섹션이 가장 중요해요. 꼼꼼히 읽어봐요.

```markdown
### 환각 검출 결과

✅ **확인됨**: Spring Boot를 활용한 RESTful API 개발 경험
   → my-profile.md: "Spring Boot 3.x 기반 백엔드 개발 (2년)"

✅ **확인됨**: MySQL 데이터베이스 설계 및 쿼리 최적화
   → my-profile.md: "MySQL, PostgreSQL 사용 경험"

⚠️ **환각 의심**: "MSA(마이크로서비스 아키텍처) 전환 프로젝트를 주도했습니다"
   → my-profile.md에서 MSA 관련 경험 확인 불가
   → 대체 제안: "모놀리식 구조의 서비스를 개발하며 확장성에 대한 고민을 했습니다"
     (my-profile.md의 실제 경험을 활용하세요)

✅ **확인됨**: Docker를 활용한 컨테이너 환경 구성
   → my-profile.md: "Docker 기반 로컬 개발 환경 구성 경험"

⚠️ **환각 의심**: "일 평균 10만 건의 트래픽을 처리하는 시스템을 운영했습니다"
   → my-profile.md에서 해당 트래픽 수치 확인 불가
   → 실제 경험한 수치로 교체하거나, 수치 없이 서술하는 것을 권장합니다
```

환각이 두 개 발견됐어요. MSA 경험과 트래픽 수치가 프로필에서 확인되지 않았어요.

### 상세 피드백과 수정 제안 확인

점수표와 환각 검출 외에도 기준별 상세 피드백과 구체적인 수정 제안이 함께 출력돼요.

```markdown
### 기준별 상세 피드백

**구체성 (12/20)**
전반적으로 경험을 나열하는 데 그치고 있어요. "API를 개발했습니다" 보다는
"주문 처리 API 5개를 설계하고, 응답 시간을 기존 대비 30% 단축했습니다"처럼
무엇을, 얼마나 했는지 수치로 보여주세요.

**차별화 (11/20)**
"팀원들과 협력하여", "문제를 해결하기 위해 노력했습니다" 같은 표현이 3회
등장해요. 이런 표현은 모든 지원자가 써요. 이 회사, 이 직무에만 해당하는
본인만의 이유를 찾아서 써주세요.

### 수정 제안 (우선순위 순)

1. **환각 제거 (필수)**: MSA 경험과 트래픽 수치를 삭제하고,
   my-profile.md에 실제로 있는 경험으로 대체하세요.

2. **구체성 강화**: "API 개발" → "결제 모듈 API 3개 설계 및 구현,
   단위 테스트 커버리지 80% 달성"처럼 수치와 맥락을 추가하세요.

3. **뻔한 표현 제거**: "노력했습니다", "최선을 다했습니다"를
   구체적인 행동 동사로 교체하세요.

4. **JD 키워드 추가**: JD에서 강조한 "대용량 데이터 처리"와
   "성능 최적화"가 자소서에서 약하게 다뤄지고 있어요.

5. **마무리 문장 강화**: 마지막 문장이 "열심히 하겠습니다"로 끝나고 있어요.
   이 회사에서 구체적으로 어떤 기여를 하고 싶은지로 마무리하세요.
```

## 반복 개선 워크플로우

한 번 검토로 끝내지 않아요. 피드백을 반영해서 수정하고, 다시 검토하는 사이클이 핵심이에요.

### 피드백을 바탕으로 수정 요청하기

검토 결과를 받았으면, 클로드에게 수정을 요청해요. 새로운 대화를 시작하지 않아도 돼요. 바로 이어서 말하면 돼요.

```bash
방금 받은 리뷰 피드백을 반영해서 자소서를 다시 써줘.
환각으로 표시된 두 항목은 my-profile.md에 실제로 있는 경험으로 교체하고,
구체성 점수를 높이기 위해 수치를 추가해줘.
수정된 자소서는 output/resume-techcorp-backend-v2.md로 저장해줘.
```

클로드가 피드백을 반영해서 수정본을 만들어줘요. 그러면 수정본으로 다시 검토를 실행해요.

```bash
/review-resume output/resume-techcorp-backend-v2.md data/analysis/techcorp-backend.json
```

### generate → review → 수정 → review 사이클

이 과정을 사이클로 정리하면 이래요.

```
/generate-resume → /review-resume → 수정 요청 → /review-resume → ...
```

반복할수록 점수가 올라가요. 처음 생성된 자소서가 60-70점대인 경우가 많아요. 대부분의 개선은 첫 번째 피드백 반영에서 이뤄지는데, 환각 제거와 수치 추가만으로도 10-15점이 올라요.

> **TIP:** 보통 2-3회 반복하면 80점 이상에 도달해요. 3회 이상 반복해도 점수가 정체된다면, 자소서 구조를 바꾸거나 강조하는 경험을 완전히 교체해보세요.

### 수정 요청을 잘 하는 방법

수정을 요청할 때도 구체적으로 말할수록 좋아요. 막연하게 "더 좋게 고쳐줘"보다는 피드백 항목을 직접 지목하는 편이 훨씬 나아요.

**덜 효과적인 요청:**
```bash
자소서를 더 좋게 수정해줘.
```

**효과적인 요청:**
```bash
리뷰 결과를 반영해서 자소서를 수정해줘.
우선순위대로 적용해줘:
1. ⚠️ 환각으로 표시된 MSA 경험과 트래픽 수치 제거
2. 구체성 항목 피드백 반영 — "API 개발" 부분에 수치 추가
3. "노력했습니다", "최선을 다했습니다" 표현 모두 제거
수정 후 output/resume-techcorp-backend-v2.md로 저장해줘.
```

피드백 항목을 번호로 지목하면 클로드가 무엇을 수정해야 하는지 명확하게 알아요. 수정 범위가 좁고 구체적일수록 의도치 않은 부분이 바뀌는 일도 줄어들어요.

### Before/After 비교

실제로 이 사이클을 돌리면 어떻게 변하는지 볼게요.

**1차 검토 결과 (초안)**

```markdown
총점: 65/100

주요 문제:
- 환각 2건 (MSA 경험, 트래픽 수치)
- 구체성 부족 (수치 없는 서술)
- 뻔한 표현 다수
```

**피드백 반영 후 2차 검토 결과**

```markdown
총점: 82/100

개선 사항:
- 환각 0건 ✅
- 구체성 12점 → 17점 (수치 3개 추가)
- 차별화 11점 → 15점 (뻔한 표현 제거)
- JD 매칭도 유지
```

17점이 올랐어요. 환각도 모두 제거됐고, 수치도 추가됐어요. 이게 사이클의 힘이에요.

> **NOTE:** 82점이 됐다고 해서 여기서 멈출 필요는 없어요. 하지만 90점 이상을 목표로 하면 자소서가 지나치게 딱딱하거나 과도하게 최적화된 느낌을 줄 수 있어요. 80-85점 수준에서 실제 면접관에게도 보여주는 것을 권장해요.

## 환각 검출이 왜 중요한가요?

지금쯤 이런 생각이 들 수 있어요. "설마 내가 쓴 자소서에 없는 내용이 있을까? AI가 그 정도는 아니지 않나?"

실제로 테스트해보면 생각보다 자주 발생해요. 특히 이런 상황에서 환각이 잘 나와요.

- JD에 있는 기술 스택이 프로필에 없을 때 (AI가 자연스럽게 끼워 맞춤)
- 경험을 "발전시켜" 쓰도록 유도할 때 (과장이 발명이 됨)
- 비슷한 경험이 있을 때 수치를 임의로 생성할 때

### 면접에서 생기는 문제

자소서에 "MSA 전환 프로젝트를 주도했다"고 썼는데, 면접관이 이렇게 물어본다면 어떨까요?

> "MSA 전환 시 서비스 간 통신은 어떻게 처리하셨나요? gRPC를 쓰셨나요, REST를 쓰셨나요?"

실제로 한 적 없는 경험이라면 대답할 수가 없어요. 당황한 모습이 그대로 드러나고, 자소서 전체의 신뢰도가 무너져요.

이건 단순히 "거짓말을 했다"의 문제가 아니에요. AI가 지어낸 내용인데, 작성자 본인도 모르고 제출했다가 면접에서 낭패를 보는 상황이에요. 기존 AI 도구를 쓴다면 이 위험을 막을 방법이 없어요.

### 우리 파이프라인이 이를 막을 수 있는 이유

`data/my-profile.md`를 **source of truth**로 명확히 설정했기 때문이에요.

클로드는 자소서를 검토할 때 "이 내용이 그럴듯한가?"를 보는 게 아니라 "이 내용이 my-profile.md에 있는가?"를 확인해요. 그럴듯함의 기준이 아니라 사실의 기준으로 검증하는 거예요.

이게 가능한 건 우리가 처음부터 프로젝트 구조를 이렇게 설계했기 때문이에요. `data/my-profile.md`를 만들고, `CLAUDE.md`에 "이 파일이 모든 자소서의 기반이다"라고 명시했고, 각 스킬이 이 파일을 참조하도록 만들었어요.

자소서 한 편을 잘 쓰는 도구가 아니라, 일관성 있고 검증된 자소서를 반복 생성할 수 있는 **파이프라인**을 만든 거예요. 그리고 그 파이프라인의 신뢰성을 보장하는 핵심이 바로 환각 검출 기능이에요.

### 지금 프로젝트 폴더 구조 확인하기

이 챕터를 마치면 프로젝트 폴더가 이렇게 구성돼 있어야 해요.

```
my-resume-pipeline/
  CLAUDE.md
  resume-v1.md
  data/
    jd/techcorp-backend.md
    analysis/techcorp-backend.json
    my-profile.md
  output/
    resume-techcorp-backend.md          ← 1차 생성 자소서
    resume-techcorp-backend-v2.md       ← 수정본
    review-resume-techcorp-backend.md   ← 검토 결과
  .claude/
    commands/
      hello.md
      collect-jd.md
      generate-resume.md
      review-resume.md                  ← 이번 챕터에서 추가
```

`output/` 폴더에 검토 결과 파일도 저장되도록 스킬에 명시했어요. 나중에 여러 회사에 지원하게 되면 각 회사별 검토 기록을 쌓아갈 수 있어요. 어떤 유형의 피드백이 반복되는지 패턴을 파악하는 데도 도움이 돼요.

> **TIP:** `data/my-profile.md`를 처음 한 번 잘 만들어두면 이후 모든 지원에서 재사용해요. 새로운 경험이 생길 때마다 프로필을 업데이트하면, 자소서 생성과 환각 검출 품질이 함께 올라가요. 프로필 관리가 파이프라인 품질의 토대가 되는 거예요.
