# JD 수집과 정리: 첫 번째 스킬 만들기

1장에서 만든 자소서, 꽤 그럴싸하죠? 그런데 솔직히 말하면 그 자소서는 "아무 회사에나 쓸 수 있는" 자소서예요. 진짜 합격하는 자소서는 달라요. JD(Job Description, 채용 공고)에 나온 키워드가 자연스럽게 녹아 있어야 해요.

취업 준비를 하면서 이런 경험 있으신가요? 자소서를 몇 시간 동안 정성껏 썼는데, 막상 제출하고 나니까 "이게 이 회사에 맞는 자소서가 맞나?" 하는 의심이 드는 것. 혹은 여러 회사에 지원하면서 같은 자소서를 조금씩 복사해서 쓰는 것. 그 방식으로는 한계가 있어요.

파이프라인의 핵심은 이거예요. 자소서를 쓰기 전에 JD를 먼저 정확하게 분석해서, 회사가 원하는 것을 데이터로 만들어두는 거예요. 그리고 내 경험 데이터와 매칭해서 맞춤형 자소서를 생성해요. 이 챕터는 그 파이프라인의 첫 번째 톱니바퀴예요.

이번 챕터에서는 JD를 체계적으로 분석해주는 첫 번째 실전 스킬을 만들어볼 거예요. 한 번 만들어두면 이후 챕터에서 자소서를 자동 생성할 때 핵심 재료로 쓰이니까, 차근차근 따라와 주세요.

이번 챕터가 끝나면 이런 것들을 할 수 있게 돼요.

- 채용 공고를 파일로 저장하고 체계적으로 관리하는 방법
- `/collect-jd` 한 줄로 어떤 JD든 6가지 항목으로 분석해서 JSON으로 저장
- 분석 결과를 자소서 작성에 실제로 활용하는 방법
- 스킬을 본인 상황에 맞게 발전시키는 방법

---

## 왜 JD 분석이 먼저인가요?

취업 준비를 하다 보면 "자소서를 어떻게 쓰냐"에 집중하기 쉬워요. 어떤 경험을 쓸까, 어떻게 표현할까. 그런데 사실 그보다 더 중요한 질문이 있어요.

**"이 회사는 무엇을 원하는가?"**

채용 담당자는 하루에도 수십 개의 자소서를 검토해요. 그분들이 가장 먼저 하는 일은 자소서에서 채용 공고에 적힌 키워드를 찾는 거예요. "Spring Boot 경험 있나? MSA 아키텍처를 언급했나? 팀 협업 이야기가 있나? 코드 리뷰 경험은?"

좋은 자소서 = JD에 나온 키워드에 딱 맞춘 자소서

이건 단순히 키워드를 가져다 붙이라는 얘기가 아니에요. 회사가 중요하게 생각하는 것들을 이해하고, 내 경험을 그 언어로 이야기하라는 거예요. 같은 경험도 어떤 언어로 포장하느냐에 따라 "딱 맞는 사람"처럼 보이기도 하고, "그냥 개발자"처럼 보이기도 해요.

그래서 자소서 파이프라인의 첫 번째 단계는 JD 분석이에요. JD에서 뽑아야 할 것은 이거예요.

- **필수 역량**: 이것만큼은 반드시 갖춰야 하는 것. 없으면 서류에서 탈락할 수 있어요.
- **우대 사항**: 있으면 좋고, 없어도 지원은 가능한 것. 비슷한 경험이 있다면 반드시 언급해요.
- **기술 스택**: 언급된 기술과 도구 (Spring, Kafka, AWS 등). 내가 아는 기술과 교집합을 찾아요.
- **핵심 키워드**: 자소서에 반드시 넣어야 할 단어들. 담당자의 눈에 바로 들어와야 해요.
- **회사 가치관**: 어떤 사람을 원하는지, 어떤 문화인지. 마지막 문단에서 이 언어를 써야 해요.

1장에서 만든 자소서는 이 분석 없이 만들어졌기 때문에 "일반적인" 자소서가 나왔어요. 앞으로는 달라질 거예요.

### JD를 직접 읽어봤을 때와 분석 후의 차이

예를 들어 테크코프 JD를 그냥 읽으면 "백엔드 개발자, Spring Boot, 경력 2~5년" 정도가 머릿속에 남아요. 그리고 자소서에 "저는 Spring Boot를 활용한 개발 경험이 있습니다"라고 쓰게 돼요.

하지만 분석 후에는 달라요. JD에서 "코드 품질과 유지보수성을 중요하게 생각하는 분"이라는 문구를 발견하고, "빠르게 변화하는 환경"이라는 키워드가 세 번 반복된다는 걸 알게 돼요. 그러면 자소서에서 이렇게 쓸 수 있어요.

> "팀에서 레거시 코드를 리팩토링하면서 테스트 커버리지를 40%에서 85%로 끌어올렸어요. 빠르게 변화하는 스타트업 환경에서도 코드 품질을 지키는 것이 장기적으로 팀의 속도를 높인다고 믿어요."

같은 경험인데, 완전히 다른 자소서가 돼요. 이게 JD 분석의 힘이에요.

## JD 저장하기

분석하려면 일단 JD가 있어야겠죠. 먼저 JD를 저장할 폴더를 만들어요.

**macOS / Linux**

```bash
mkdir -p data/jd
mkdir -p data/analysis
```

**Windows (PowerShell)**

```bash
New-Item -ItemType Directory -Force -Path data\jd
New-Item -ItemType Directory -Force -Path data\analysis
```

> **NOTE:** 터미널 명령어가 낯설다면 Claude에게 "data/jd 폴더와 data/analysis 폴더를 만들어줘"라고 요청해도 돼요. Claude 코드가 직접 터미널 명령어를 실행해준답니다.

### 채용 공고는 어디서 가져오나요?

사람인, 잡코리아, 원티드, LinkedIn, 회사 공식 채용 페이지 등에서 관심 있는 공고를 찾아요. 원하는 공고를 발견하면 텍스트를 전부 복사해서 마크다운 파일로 저장해요.

복사할 때는 이런 부분들을 꼭 포함해요.
- 회사 소개 (어떤 서비스를 만드는 회사인지)
- 담당 업무
- 필수 역량
- 우대 사항
- 기술 스택 (명시적으로 나열된 경우)
- 채용 혜택이나 문화 관련 문구

공고 페이지 URL도 파일 맨 위에 함께 메모해두면 좋아요. 나중에 면접 준비를 할 때 원본 공고를 다시 찾기가 훨씬 편해요. 채용 공고는 마감되면 사라지는 경우가 많으니까, 텍스트를 저장해두는 것 자체가 이미 중요한 작업이에요.

> **WARNING:** 웹 스크래핑(자동 수집 프로그램)으로 채용 사이트 데이터를 긁어오는 방식은 사이트 이용 약관 위반이 될 수 있어요. 법적 이슈를 피하기 위해 이 책에서는 수동으로 복사하는 방법을 사용해요.

### 파일명 규칙 정하기

나중에 파일이 많아지면 관리가 어려워져요. 처음부터 규칙을 정해두는 게 좋아요. 이 책에서는 `회사명-직무.md` 형식을 사용해요.

```
data/jd/
  techcorp-backend.md
  awesomeco-frontend.md
  bigtech-devops.md
```

직관적이고, 분석 결과 파일명과도 자연스럽게 대응돼요 (`techcorp-backend.json`).

### 연습용 예시 JD 만들기

바로 연습할 수 있도록 가상의 스타트업 **테크코프(TechCorp)** 의 채용 공고를 준비했어요. `data/jd/techcorp-backend.md` 파일을 만들고 아래 내용을 붙여 넣어요.

```markdown
# 테크코프(TechCorp) — 백엔드 개발자 채용

## 회사 소개
테크코프는 2021년 설립된 B2B SaaS 스타트업으로, 중소기업 대상 재고 관리 플랫폼을
운영하고 있어요. 현재 고객사 300개, 시리즈 A 투자 완료 단계이며 빠르게 성장 중이에요.
작은 팀이지만 뛰어난 동료들과 함께 제품을 만들어가는 경험을 소중하게 생각해요.
기술 부채를 쌓지 않고, 지속 가능한 속도로 성장하는 것이 우리의 방식이에요.

## 포지션
백엔드 개발자 (경력 2~5년)

## 담당 업무
- REST API 설계 및 개발
- 데이터베이스 설계 및 쿼리 최적화
- 서비스 성능 개선 및 장애 대응
- 코드 리뷰 및 기술 문서 작성
- 신규 기능 기획 단계부터 개발팀 의견 반영

## 필수 역량
- Java 또는 Kotlin 기반 Spring Boot 개발 경험 2년 이상
- MySQL 또는 PostgreSQL 실무 경험
- RESTful API 설계 및 구현 능력
- Git을 활용한 협업 경험
- 기술 문서 작성 및 코드 리뷰 경험

## 우대 사항
- MSA(마이크로서비스 아키텍처) 설계 및 운영 경험
- AWS, GCP 등 클라우드 서비스 사용 경험
- Redis, Kafka 등 메시지 큐/캐시 활용 경험
- 스타트업 환경에서의 개발 경험
- 테스트 코드 작성 및 CI/CD 파이프라인 구성 경험

## 기술 스택
Java, Kotlin, Spring Boot, JPA, MySQL, Redis, Docker, AWS EC2, S3, GitHub Actions

## 우리가 찾는 사람
- 문제를 정의하고 스스로 해결책을 찾아가는 분
- 동료와 적극적으로 소통하며 함께 성장하는 분
- 코드 품질과 유지보수성을 중요하게 생각하는 분
- 빠르게 변화하는 환경을 즐기는 분
- 완성도 높은 제품을 만드는 데 자부심을 느끼는 분

## 근무 조건
- 위치: 서울 강남구 (주 3회 출근, 주 2회 재택)
- 연봉: 협의 (면접 후 결정)
- 복지: 점심 제공, 도서 구매 지원 월 5만원, 컨퍼런스 참가 지원, 장비 지원
```

Claude에게 이렇게 요청하면 바로 만들어줘요.

> "`data/jd/techcorp-backend.md` 파일을 만들고, 위 내용을 저장해줘."

파일이 생성되면 프로젝트 구조가 이렇게 돼요.

```
my-resume-pipeline/
  data/
    jd/
      techcorp-backend.md   ← 방금 저장한 JD
    analysis/               ← 분석 결과가 들어갈 빈 폴더
```

## /collect-jd 스킬 만들기

자, 이제 이 JD를 분석해주는 스킬을 만들 차례예요. 2장에서 `/hello` 스킬을 만들었던 것 기억하시죠? 같은 방식으로 `.claude/commands/collect-jd.md` 파일을 만들 거예요.

### 핵심 먼저 이해하기

전체 스킬 파일을 보여주기 전에, 가장 중요한 세 가지 아이디어를 먼저 이해해봐요. 이 개념들은 앞으로 만들 모든 스킬에서 반복해서 쓰이니까 지금 익혀두면 좋아요.

**1. 역할 지정 — "당신은 채용 공고 분석 전문가입니다"**

Claude에게 역할을 명확히 지정하면 응답의 품질이 달라져요. "JD 분석해줘"라고 하는 것보다 "채용 공고 분석 전문가로서 분석해줘"라고 하면 더 정확하고 구체적인 결과가 나와요. 프롬프트 엔지니어링의 기본 중의 기본이에요.

왜 이게 효과가 있냐고요? Claude는 엄청난 양의 텍스트를 학습했어요. "채용 공고 분석 전문가"라는 역할을 부여하면, Claude는 그 역할에 맞는 사고 패턴과 언어를 활성화해요. 인사 담당자가 JD를 읽는 방식, 헤드헌터가 포지션을 분석하는 방식 — 그런 관점에서 응답해줘요.

**2. `$ARGUMENTS` — 사용자 입력을 자동으로 받아요**

`/collect-jd data/jd/techcorp-backend.md` 라고 입력하면, 스킬 파일 안의 `$ARGUMENTS` 자리에 `data/jd/techcorp-backend.md` 가 자동으로 들어가요. Claude 코드가 해주는 일종의 변수 치환이에요.

덕분에 스킬 하나로 어떤 JD 파일이든 처리할 수 있어요. 파일 경로를 바꿔서 입력하면 그때그때 다른 JD를 분석해줘요.

**3. JSON으로 저장 — 다음 단계에서 쓸 재료를 만들어요**

분석 결과를 그냥 텍스트로 출력하면 나중에 활용하기 어려워요. 사람이 읽기는 좋지만, 프로그램이나 다른 프롬프트가 읽기는 어렵거든요. JSON 형식으로 저장해두면 5장에서 자소서를 자동 생성할 때 이 데이터를 읽어서 활용할 수 있어요.

JSON을 직접 다뤄본 적 없어도 괜찮아요. 우리는 만드는 방법만 알면 되고, 나머지는 Claude가 다 해줘요.

### 전체 스킬 파일 작성

이제 실제 파일을 만들어볼게요. Claude 코드에게 이렇게 요청해요.

> "`.claude/commands/collect-jd.md` 파일을 만들어줘. 아래 내용을 그대로 넣어줘."

~~~markdown
`.claude/commands/collect-jd.md` 파일 전체 내용:

```markdown
당신은 채용 공고(JD) 분석 전문가입니다.

### 입력
$ARGUMENTS 에 JD 파일 경로가 주어집니다.
해당 파일을 읽어주세요.

### 분석 항목
다음 항목을 JD에서 추출하세요:
1. 회사명과 직무명
2. 필수 역량 — 목록으로
3. 우대 사항 — 목록으로
4. 기술 스택 — 언급된 기술/도구 모두
5. 핵심 키워드 — 자소서에 반드시 포함해야 할 단어 10개
6. 회사 가치관/문화

### 출력
분석 결과를 data/analysis/ 폴더에 JSON 파일로 저장하세요.
파일명은 JD 파일명과 동일하게 해주세요 (예: techcorp-backend.json).

JSON 구조:
{
  "company": "회사명",
  "position": "직무명",
  "required": ["필수역량1", "필수역량2"],
  "preferred": ["우대사항1", "우대사항2"],
  "tech_stack": ["기술1", "기술2"],
  "keywords": ["키워드1", "키워드2", ...10개],
  "culture": "회사 가치관/문화 요약"
}

저장 완료 후 저장된 파일 경로와 추출된 핵심 키워드 10개를 화면에 출력해주세요.
```
~~~

파일이 저장됐으면 각 섹션이 어떤 역할을 하는지 살펴볼게요.

**역할 지정 부분**: 첫 줄 "당신은 채용 공고(JD) 분석 전문가입니다"가 Claude의 페르소나를 설정해요. 이 한 줄이 있고 없고의 차이가 생각보다 커요. 나중에 직접 비교해보면 재미있을 거예요.

**`$ARGUMENTS` 입력 부분**: "해당 파일을 읽어주세요"라는 지시와 함께 파일 경로를 받아요. Claude는 그 경로의 파일을 실제로 열어서 내용을 읽어요. 파일이 없으면 없다고 알려줄 거예요.

**분석 항목 6가지**: 번호를 매겨서 무엇을 추출할지 명확히 지시해요. "핵심 키워드 10개"처럼 숫자를 명시하면 결과가 일관돼요. "적당히 뽑아줘"라고 하면 3개 나올 때도 있고 15개 나올 때도 있어요.

**JSON 출력 형식**: 구체적인 키 이름(`company`, `position`, `required` 등)과 구조를 지정했어요. 이 필드명들은 이후 챕터에서 자소서 생성 스킬이 정확히 이 이름으로 읽어서 사용하게 돼요. 미리 약속을 정해두는 셈이에요.

**저장 경로 지정**: `data/analysis/` 폴더에 저장하도록 했어요. 파일명은 JD 파일명과 맞추면 나중에 어떤 JD를 분석한 결과인지 한눈에 알 수 있어요.

### 결과를 어떻게 활용하나요?

각 필드가 무엇을 의미하는지, 그리고 자소서 작성에 어떻게 연결되는지 살펴볼게요.

**`required` — 합격의 기준선**

이 항목에 해당하는 경험이 있다면 자소서에서 꼭 강조해야 해요. 예를 들어 "Spring Boot 개발 경험 2년 이상"이 요구된다면, 자소서에서 "Spring Boot를 사용해서 ○○ 시스템을 구축했어요"처럼 명확하게 써야 해요. 없는 항목은 솔직하게 인정하되, 관련 경험이 있다면 그걸로 연결해봐요.

**`preferred` — 차별점을 만들 기회**

우대 사항은 모든 지원자가 갖추기 어려운 것들이에요. 여기 해당하는 경험이 하나라도 있다면 반드시 언급해요. "MSA 경험은 없지만 모놀리식 → 서비스 분리 경험이 있어요" 식으로 연결하는 것도 좋아요.

**`tech_stack` — 교집합 찾기**

내가 써본 기술과 교집합을 찾아요. 교집합이 많을수록 이 포지션에 적합하다는 뜻이에요. 전혀 교집합이 없다면 지원 전에 한 번 더 고민해보는 게 좋아요.

**`keywords` — 자소서의 언어를 맞춰요**

5장에서 자소서를 생성할 때 이 키워드들이 자연스럽게 자소서에 포함될 거예요. 담당자가 읽을 때 "맞아, 이런 사람이 필요했어"라는 느낌을 받아야 해요.

**`culture` — 마지막 문단의 핵심**

자기소개서의 마지막 문단(지원 동기, 입사 후 포부 등)을 쓸 때 이 문화에 맞는 언어를 사용해봐요. "빠르게 성장하는 환경에서 동료와 함께 완성도 높은 제품을 만들고 싶어요"처럼 회사의 언어로 이야기하면 훨씬 인상이 달라져요.

테크코프의 경우 culture 필드에 "자기 주도적으로 문제를 정의하고 해결책을 찾는다"는 내용이 담겼어요. 그러면 자소서에서 이렇게 쓸 수 있어요.

> "새로운 기능을 개발할 때 단순히 티켓을 처리하는 게 아니라, 사용자가 실제로 겪는 문제가 무엇인지부터 파악해요. 요구사항에 의문이 생기면 먼저 관련 데이터를 찾아보고, 팀원들과 함께 더 좋은 방향을 논의하는 걸 즐겨요."

똑같은 말을 하더라도, 회사가 쓰는 언어("문제를 정의하고 스스로 해결책을 찾는다")와 맞닿아 있으면 담당자가 공감하기 훨씬 쉬워요.

> **TIP:** 분석 결과가 마음에 들지 않거나 빠진 항목이 있다면, Claude에게 바로 요청해봐요. "keywords에 '쿼리 최적화'도 추가해줘", "culture 부분을 더 구체적으로 써줘" 식으로요. Claude는 기존 파일을 수정해서 업데이트된 버전을 저장해줘요.

## 여러 JD 분석해보기

한 번 만들어두니까 계속 쓸 수 있다는 게 바로 스킬의 가치예요. 테크코프 JD 하나로는 충분히 못 느낄 수 있으니, 실제로 관심 있는 공고를 하나 더 가져와서 분석해봐요.

예를 들어 다른 회사의 프론트엔드 개발자 공고를 `data/jd/awesomeco-frontend.md` 로 저장했다면 이렇게 하면 돼요.

```bash
/collect-jd data/jd/awesomeco-frontend.md
```

같은 명령어, 다른 파일. 결과는 `data/analysis/awesomeco-frontend.json` 으로 저장돼요.

스킬을 수정하지 않아도, 설치하지 않아도 — 그냥 파일 경로만 바꿔서 입력하면 돼요. 이게 `$ARGUMENTS` 의 힘이에요.

### 어떤 JD를 골라야 할까요?

막연하게 "좋아 보이는 곳"에 지원하면 지치기 쉬워요. 파이프라인을 본격적으로 활용하기 전에, JD를 고르는 기준을 잠깐 생각해봐요.

**기술 스택 교집합 50% 이상**: tech_stack 필드와 내가 아는 기술을 비교해봐요. 절반 이상 겹치는 공고가 현실적인 지원 대상이에요. 교집합이 20% 미만이라면 준비 기간이 필요해요.

**필수 역량의 현실성**: required 필드를 보면서 "이중 내가 설명할 수 있는 게 몇 개인가?" 를 세어봐요. 5개 중 4개 이상이면 좋은 신호예요.

**회사 규모와 성장 단계 고려**: JD만 봐도 어느 정도 회사 규모가 느껴져요. 스타트업은 "빠른 의사결정", "다양한 역할"을 강조하고, 대기업은 "프로세스", "조직 문화"를 강조하는 경향이 있어요. 본인이 어떤 환경을 원하는지 먼저 결정해두면 JD 고르기가 훨씬 쉬워요.

### 여러 JD를 한꺼번에 분석하고 싶다면

JD 파일이 많아지면 하나하나 실행하기 귀찮아질 수 있어요. 그럴 때는 Claude에게 이렇게 요청하면 돼요.

> "data/jd/ 폴더에 있는 모든 JD 파일을 분석해줘. 각각에 대해 /collect-jd 스킬과 동일한 방식으로 분석하고 data/analysis/ 폴더에 저장해줘."

Claude는 폴더를 스캔하고 각 파일을 순서대로 분석해서 저장해줘요. 스킬을 여러 번 실행하는 수고를 덜 수 있어요.

분석이 끝나면 이렇게 요청해서 한눈에 비교할 수도 있어요.

> "data/analysis/ 폴더의 모든 JSON 파일을 읽고, 회사별로 tech_stack을 비교해줘. 어떤 기술이 가장 많이 요구되는지 정리해줘."

이렇게 하면 여러 회사의 JD에서 공통으로 등장하는 기술 스택을 파악할 수 있어요. "Spring Boot가 5개 중 4개 공고에 나왔다"는 걸 알면, 그 기술을 더 깊이 준비해야 한다는 신호예요.

> **TIP:** 관심 있는 공고를 5~10개 모아두면 나중에 엄청난 무기가 돼요. 6장에서는 저장된 JSON 분석 결과를 읽어서 각각에 맞는 자소서를 자동으로 생성할 거예요. 지금 JD를 많이 모아둘수록 나중에 할 일이 줄어들어요.

### 잘못됐을 때 어떻게 하나요?

실행하다 보면 가끔 이런 문제가 생겨요.

**파일을 못 찾는 경우**: 파일 경로를 확인해봐요. 현재 프로젝트 폴더(`my-resume-pipeline/`)에서 실행하고 있는지, 경로가 정확한지 확인해요. Claude에게 "현재 폴더 구조를 보여줘"라고 하면 어디서 실행 중인지 확인할 수 있어요.

**JSON 구조가 이상한 경우**: Claude에게 "결과를 다시 확인하고 JSON 형식에 맞게 수정해줘"라고 요청해봐요. 혹은 스킬을 다시 실행하면 파일을 덮어씌워요.

**분석 내용이 부정확한 경우**: 대화를 통해 다듬어요. "keywords가 너무 일반적인 것 같아. JD에서 더 구체적인 기술 키워드 위주로 다시 추출해줘"라고 하면 Claude가 파일을 다시 읽고 업데이트해줘요.

**스킬 명령어가 인식되지 않는 경우**: `.claude/commands/collect-jd.md` 파일이 올바른 위치에 있는지 확인해요. Claude에게 "`.claude/commands/` 폴더 안에 어떤 파일이 있어?"라고 물어보면 현재 등록된 스킬 목록을 알려줘요.

> **NOTE:** 스킬 파일 자체를 수정하고 싶다면 `.claude/commands/collect-jd.md` 를 직접 열어서 편집하면 돼요. 예를 들어 키워드 개수를 10개에서 15개로 늘리고 싶다면 "핵심 키워드 — 자소서에 반드시 포함해야 할 단어 10개" 부분을 15개로 바꾸면 다음 실행부터 적용돼요. 스킬 파일은 그냥 텍스트 파일이니까 언제든지 수정할 수 있어요.

## 스킬을 개선해보기

스킬을 한 번 써보면 "이건 좀 더 있으면 좋겠는데" 하는 게 생기기 마련이에요. 그게 자연스러운 과정이에요. 스킬 파일은 언제든 수정할 수 있으니까, 필요에 따라 발전시켜 나가면 돼요.

### 분석 언어 수준 지정하기

같은 JD라도 어떤 수준에서 분석하느냐에 따라 결과가 달라질 수 있어요. 더 깊이 있는 분석을 원한다면 스킬 파일에 이런 지시를 추가해봐요.

```markdown
### 분석 심도
- 명시적으로 적힌 내용뿐 아니라, JD의 행간에서 읽히는 회사의 니즈도 파악해주세요.
- 예: "빠르게 변화하는 환경"이라는 문구가 있다면, 이 회사는 안정보다 성장을 중시한다는 의미예요.
```

이렇게 한 줄을 추가하는 것만으로도 분석 품질이 달라져요.

### 적합도 점수 추가하기

여러 JD를 분석하다 보면 "이 회사가 나한테 맞나?" 를 빠르게 판단하고 싶어질 거예요. 그럴 때는 JSON에 `fit_score` 필드를 추가해볼 수 있어요.

스킬 파일의 JSON 구조 부분을 이렇게 바꿔요.

```markdown
JSON 구조:
{
  "company": "회사명",
  "position": "직무명",
  "required": ["필수역량1", "필수역량2"],
  "preferred": ["우대사항1", "우대사항2"],
  "tech_stack": ["기술1", "기술2"],
  "keywords": ["키워드1", ...10개],
  "culture": "회사 가치관/문화 요약",
  "fit_score": {
    "score": 0,
    "reason": "점수 산정 이유"
  }
}

fit_score는 0~10 사이의 정수로, JD에서 요구하는 역량과 기술 스택이 얼마나
일반적인 백엔드 개발자 프로파일과 부합하는지를 나타내요.
점수 기준: 8-10 (매우 적합), 5-7 (적합), 3-4 (부분 적합), 0-2 (부적합)
```

물론 "일반적인 백엔드 개발자 프로파일" 부분은 본인 상황에 맞게 바꿔야 해요. 4장에서 경험 DB를 만들고 나면, 그 데이터를 여기에 참조하도록 업그레이드할 수도 있어요.

### 분석 결과를 마크다운으로도 저장하기

JSON은 기계가 읽기 편하지만, 사람이 빠르게 훑어보기엔 마크다운이 더 좋아요. 두 형식 모두 저장하고 싶다면 스킬 파일에 이렇게 추가해봐요.

```markdown
### 추가 출력
JSON 파일 저장 외에, 동일한 분석 내용을 읽기 쉬운 마크다운 형식으로
data/analysis/{파일명}-summary.md 에도 저장해주세요.
마크다운에는 각 항목별 간략한 설명과 자소서 작성 시 유의점을 포함해주세요.
```

이렇게 하면 `techcorp-backend.json` 과 `techcorp-backend-summary.md` 두 파일이 생겨요. JSON은 나중에 자소서 생성 스킬에서 읽고, 마크다운은 직접 눈으로 훑어볼 때 사용하면 돼요.

> **TIP:** 스킬을 수정할 때는 조금씩 바꾸고, 바꿀 때마다 한 번 실행해서 결과를 확인해봐요. 한꺼번에 많이 바꾸면 무엇이 문제인지 찾기 어려워요. 작은 변화 → 확인 → 다음 변화의 리듬이 좋아요.

## 정리

이번 챕터에서 한 것들을 정리해볼게요.

**JD 분석이 왜 중요한지** 이해했어요. 합격하는 자소서는 JD 키워드에 맞춰 쓰여야 해요. 같은 경험이라도 회사의 언어로 이야기하면 "딱 맞는 사람"이 되고, 그냥 쓰면 "평범한 지원자"가 돼요.

**JD 저장 폴더 구조**를 만들었어요. `data/jd/`에 원본 JD를, `data/analysis/`에 분석 결과를 저장하는 방식이에요. 파일명은 `회사명-직무.md` 규칙으로 정리해요.

**`/collect-jd` 스킬**을 만들었어요. 이 과정에서 세 가지 핵심 패턴을 배웠어요.
- **역할 지정**: "당신은 전문가입니다" 한 줄이 응답 품질을 바꿔요
- **`$ARGUMENTS`**: 파일 경로를 인자로 받아서 스킬 하나로 여러 JD를 처리해요
- **JSON 저장**: 구조화된 데이터로 저장해야 다음 스킬에서 활용할 수 있어요

이 세 가지 패턴은 앞으로 만들 모든 스킬에서 반복해서 쓰여요. 지금 잘 익혀두면 이후 챕터가 훨씬 쉬워질 거예요.

**실제 JD를 분석**해봤어요. 테크코프 예시로 스킬이 제대로 동작하는 것을 확인했고, 결과 JSON의 각 필드(`required`, `preferred`, `tech_stack`, `keywords`, `culture`)가 자소서 작성에 어떻게 연결되는지도 살펴봤어요.

**스킬을 발전시키는 방법**도 알아봤어요. 분석 심도 추가, 적합도 점수, 마크다운 요약 파일 생성 — 필요에 따라 스킬 파일을 수정하면서 본인 상황에 맞게 키워나갈 수 있어요.

다음 챕터에서는 이 JSON 분석 결과를 바탕으로 나의 경험 데이터베이스를 만들어볼 거예요. JD가 "회사가 원하는 것"이라면, 경험 DB는 "내가 가진 것"이에요. 이 두 가지를 매칭하는 게 자소서 파이프라인의 핵심이에요. 두 재료가 모이면, 비로소 진짜 나만의 자소서가 만들어져요.

지금 당장 관심 있는 공고 하나를 찾아서 `data/jd/` 폴더에 저장해봐요. 다음 챕터를 시작하기 전에 JD를 최소 한 개 이상 모아두는 게 좋아요.
