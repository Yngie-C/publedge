# 프로젝트 섹션 추가: 카드 3장 그리드

Ch1에서 명함 페이지 하나가 화면에 떴어요. 이름과 사진, 한 줄 소개가 담긴 카드 한 장이 화면 정중앙에 자리를 잡았죠. 이제 그 아래에 프로젝트 카드 3장을 나란히 놓아볼게요. 대화 한 번으로 페이지에 내 이야기가 한 층 더 쌓입니다.

포트폴리오가 '명함'에서 '포트폴리오'로 달라지는 순간은 바로 이 챕터예요. 이름과 얼굴만 있는 페이지는 누구나 만들 수 있지만, 내가 무엇을 만들었는지 보여주는 카드가 붙는 순간 방문자는 "이 사람은 이런 걸 했구나"를 바로 읽을 수 있게 됩니다. 채용 담당자가 포트폴리오를 보는 시간은 평균 30초 남짓이에요. 카드 3장이 그 30초를 채워줍니다.

## 이번 챕터의 목표

- 내 프로젝트 3개를 한 줄 설명과 링크로 정리한다
- 명함 아래에 프로젝트 카드 3장 그리드를 추가한다
- 카드를 클릭하면 새 탭에서 링크가 열리도록 요청한다

이번 챕터에서는 코드를 보는 시간보다 내 프로젝트를 정리하는 시간이 더 길어요. 그게 정상입니다. 무엇을 만들었는지 말할 수 있어야 Claude도, 방문자도 이해할 수 있거든요. 준비가 잘 될수록 프롬프트 한 번에 원하는 결과가 나올 확률이 높아집니다.

이번 챕터를 통해 "말로 UI를 만든다"는 감각이 조금 더 구체적으로 느껴지길 바라요. HTML 태그 이름을 몰라도, CSS 속성을 외우지 않아도, 내가 원하는 결과를 언어로 표현할 수 있으면 충분해요. 바이브 코딩의 핵심 역량은 코딩 실력이 아니라 명확하게 요청하는 능력이에요.

## 내 프로젝트 3개 정리하기

Claude에게 프로젝트 카드를 만들어달라고 할 때, 미리 정보를 준비해두면 훨씬 빠르게 진행돼요. 각 프로젝트마다 세 가지만 정리하면 됩니다. **제목**, **한 줄 설명**, **링크**(GitHub 저장소, Notion 페이지, 배포 URL 등 아무거나).

한 줄 설명은 짧을수록 좋아요. "Python으로 데이터를 분석했습니다"보다 "공공데이터 API로 서울시 카페 밀집도를 지도에 시각화"처럼 무엇을 어떻게 만들었는지가 담긴 문장이 훨씬 강하게 남아요. 기술 스택을 나열하기보다 결과물이 무엇인지를 중심으로 한 문장을 짜보세요.

프로젝트 정보를 Claude에게 전달할 때 형식보다 내용이 중요해요. 위 표처럼 정리해서 붙여넣어도 되고, 그냥 "프로젝트 1은 제목이 '서울 카페 데이터 분석'이고, 설명은 '공공데이터 API로 서울시 카페 밀집도를 시각화'이고, 링크는 https://github.com/minji/cafe-map 이에요"처럼 문장으로 말해도 Claude가 잘 파악해요. 표 형식이 편하면 표를, 문장이 편하면 문장으로 주면 됩니다.

아직 링크가 없으면 `#`이나 임시 URL을 써도 괜찮아요. 나중에 말로 고치면 됩니다.

아래는 취준생 독자를 위한 예시 3개예요. 본인 상황에 맞게 제목과 설명을 바꿔 쓰면 됩니다. 예시 그대로 넣고 나중에 고쳐도 괜찮아요.

| 제목 | 한 줄 설명 | 링크 |
|------|-----------|------|
| 서울 카페 데이터 분석 | 공공데이터 API로 서울시 카페 밀집도를 시각화한 토이 프로젝트 | `https://github.com/minji/cafe-map` |
| 뉴스 클론 코딩 | 네이버 뉴스 메인 페이지를 HTML/CSS로 클론한 연습 프로젝트 | `https://github.com/minji/news-clone` |
| 동아리 홈페이지 | 데이터 분석 동아리 소개 및 활동 기록을 담은 정적 사이트 | `https://minji-club.vercel.app` |

> **TIP:** 아직 공개할 프로젝트가 없어도 괜찮아요. '진행 중인 것'이라도 카드로 올려두세요. "프로젝트 1의 설명을 ___로 바꿔주세요"라고 말하면 언제든지 고칠 수 있어요.

프로젝트가 세 개가 안 된다면 어떻게 할까요? 학교 수업 과제, 개인 스터디 기록, 공모전 출품작도 카드에 올릴 수 있어요. 완성도보다 "내가 뭔가를 만들어봤다"는 흔적이 중요합니다. 카드 한 장에 "진행 중"이라고 표시해두는 것도 방법이에요. 방문자는 완성된 결과물만 보고 싶어하는 게 아니라, 이 사람이 어떤 방향으로 성장하고 있는지를 함께 보거든요.

## 프롬프트 쓰기

아래 프롬프트를 복사한 뒤 프로젝트 정보를 채워 넣으세요.

```markdown
index.html 아래에 "프로젝트" 섹션을 추가해 주세요.
카드 3장을 가로 그리드로 배치하고, 각 카드는 제목·설명·"자세히 보기" 링크를 담아요.
카드 링크는 새 탭에서 열리게 해주세요 (target="_blank").
명함은 그대로 두고, 명함 아래에만 추가해 주세요.
프로젝트 1: [제목] / [설명] / [링크]
프로젝트 2: [제목] / [설명] / [링크]
프로젝트 3: [제목] / [설명] / [링크]
```

프롬프트에서 두 표현이 핵심이에요.

`"명함은 그대로 두고, 명함 아래에만 추가해 주세요"`라는 문장이 첫 번째입니다.

> **NOTE:** 이렇게 명시적으로 말하면 Claude가 기존 코드를 건드리지 않고 새 섹션만 덧붙여요. 반대로 "프로젝트 섹션 만들어줘"처럼 짧게 말하면 명함 스타일을 통째로 바꾸거나 파일 구조를 재편할 수 있어요. 보존하고 싶은 부분은 항상 명시적으로 말해주세요.

"어디까지 건드려도 된다"는 범위를 명확히 말하는 습관은 바이브 코딩 전반에서 계속 쓰이는 패턴이에요. 지금 당장은 명함을 보호하는 용도지만, 나중에 특정 섹션만 남기고 나머지를 재편하거나, 모바일 스타일만 건드리고 데스크탑은 그대로 두는 요청에도 같은 방식이 적용돼요. "이 부분은 건드리지 말아요"라는 한 마디가 Claude의 작업 범위를 정확히 잡아줍니다.

프롬프트를 보내기 전에 30초만 읽어보는 습관도 도움이 돼요. "내가 지금 뭘 원하는지"가 명확하면 Claude의 응답도 명확해져요. 반대로 "뭔가 추가해줘"처럼 모호한 요청은 Claude가 임의로 해석해서 예상 밖의 결과를 낼 수 있어요. 이번 프롬프트 템플릿처럼 **무엇을**, **어디에**, **어떻게**, **건드리지 말아야 할 부분**을 담은 구조가 가장 안정적이에요.

두 번째는 `target="_blank"` 언급이에요. 이 한 마디가 링크 동작을 결정합니다. 빠뜨리면 카드를 클릭했을 때 같은 탭에서 페이지가 이동해 포트폴리오 화면이 사라져요.

> **WARNING:** 가끔 Claude가 `index2.html`이나 `projects.html` 같은 **새 파일을 만들 때가 있어요**. 응답에서 수정된 파일 이름이 `index.html`인지 꼭 확인하세요. 다른 이름이면 "꼭 index.html에 추가해 주세요. 새 파일은 만들지 말아요"라고 다시 말하면 됩니다. 이 증상은 부록 A에서도 다뤄요.

## Claude가 만든 변경사항 확인하기

Claude가 응답을 마치면 `index.html`을 저장하고 브라우저에서 새로고침해 보세요. 저장 단축키는 macOS라면 `Cmd+S`, Windows라면 `Ctrl+S`예요. 저장하지 않으면 브라우저가 이전 버전을 보여주므로, 새로고침 전에 저장을 먼저 확인하세요.

명함이 위에 있고, 그 아래에 "프로젝트" 제목과 함께 카드 3장이 가로로 나란히 배치돼 있다면 성공입니다.

한 가지 눈여겨볼 점이 있어요. Ch1에서는 명함이 화면 정중앙에 떠 있었어요. `body`에 `justify-content: center`가 걸려서 수직으로도 가운데 정렬되어 있었거든요. 그런데 이번 결과를 보면 명함이 위쪽으로 올라가 있고, 그 아래에 프로젝트 섹션이 이어지는 구조로 바뀌어 있을 거예요.

이건 Claude가 프로젝트 섹션을 붙이면서 `body`의 `flex-direction`을 `row`에서 `column`으로, `justify-content`를 `center`에서 `flex-start`로 조정했기 때문이에요. 명함 하나만 있을 때는 수직 가운데 정렬이 자연스럽지만, 콘텐츠가 두 블록 이상이 되면 위에서 아래로 쌓이는 흐름이 훨씬 자연스럽거든요. Claude가 맥락을 읽고 레이아웃 방향을 스스로 바꾼 거예요.

바이브 코딩에서 레이아웃을 말로 고친다는 게 바로 이런 감각이에요. "명함 아래에 카드를 붙여줘"라는 요청 한 마디가 단순히 HTML 엘리먼트를 추가하는 것에 그치지 않아요. Claude는 그 말을 듣고 "아, 콘텐츠가 세로로 쌓여야 하는 구조구나"를 자동으로 판단해서 CSS까지 함께 조정해줍니다. 결과가 어색하다면 "명함을 다시 화면 정중앙에 놓되, 아래에 프로젝트 섹션이 이어지게 해주세요"처럼 원하는 배치를 말로 정확히 설명하면 됩니다.

이 자동 조정이 때로는 의도치 않은 부분까지 건드리기도 해요. "명함 배경색이 바뀐 것 같아요. 원래대로 흰색으로 돌려주세요"처럼 변한 부분을 콕 집어 말하면 됩니다. Claude는 전체를 다시 짜는 대신 해당 속성만 되돌려줘요. 이렇게 조금씩 교정해가는 과정이 바이브 코딩의 실제 작업 흐름이에요. 한 번에 완벽할 필요가 없어요.

카드 3장이 균등하게 배치되고, 각 카드에 프로젝트 제목·설명·"자세히 보기" 링크가 들어있으면 이번 챕터의 핵심은 이미 완성된 거예요.

카드 너비가 고르지 않거나 세로로 쌓여 보인다면 창 너비 문제일 수 있어요. 브라우저 창을 넓게 펼쳐보세요. 그래도 이상하다면 "카드 3장이 가로 한 줄에 균등하게 배치되지 않아요. 다시 잡아주세요"라고 말하면 됩니다. `grid-template-columns: repeat(3, 1fr)` 이 한 줄이 그리드를 잡아주는데, 이 부분이 빠진 경우에 주로 발생해요.

모바일에서 확인해보면 카드 3장이 세로로 쌓여 보일 수 있어요. 이건 반응형 처리가 아직 안 된 상태라 정상이에요. Ch4에서 모바일 레이아웃을 별도로 다루기 때문에 지금은 데스크탑 화면만 확인하면 돼요. 스마트폰으로 보면 카드가 좁아 보이거나 글씨가 잘릴 수 있는데, 지금 단계에서는 신경 쓰지 않아도 됩니다.

## 링크 동작 테스트

"자세히 보기" 링크를 클릭해 보세요. 새 탭이 열리면서 해당 URL로 이동하면 정상입니다.

새 탭이 열리지 않는다면 `target="_blank"`가 빠진 거예요. Claude에게 "카드 링크가 새 탭에서 열리지 않아요. `target=\"_blank\"`를 추가해 주세요"라고 말하면 됩니다.

링크가 `#`이면 클릭해도 이동하지 않는 게 정상이에요. 실제 URL로 바꾸면 그때 새 탭이 열립니다.

링크 테스트는 포트폴리오를 공개하기 전에 반드시 해야 할 마지막 점검이에요. 취업 면접에서 "포트폴리오 링크 보내주세요"라는 말을 들었을 때, 막상 클릭해보니 링크가 깨져 있거나 같은 탭에서 페이지가 사라지면 인상이 나빠질 수 있어요. 1분이면 확인 끝납니다. 반드시 직접 클릭해보세요.

GitHub 저장소 링크를 쓸 때는 저장소가 **Public**으로 설정되어 있는지 확인하세요. Private 저장소는 링크를 눌러도 404 오류나 로그인 화면이 나와요. 저장소 공개 여부는 GitHub 저장소 설정(Settings)에서 바꿀 수 있어요. 포트폴리오에 올릴 프로젝트라면 Public으로 전환해두는 게 좋아요.

링크가 올바른지 확인하는 가장 빠른 방법은 시크릿 모드(Incognito)에서 열어보는 거예요. 내 계정으로 로그인된 상태에서는 Private 저장소도 보이기 때문에 착각하기 쉬워요. 시크릿 모드는 로그인 상태가 없어서 방문자 입장에서 링크가 제대로 열리는지 정확하게 확인할 수 있어요. 포트폴리오를 공개하기 전에 시크릿 모드 테스트를 습관으로 들여두면 좋아요.

## 내용 고쳐보기

카드가 화면에 나왔다면 이번엔 내용을 바꿔볼 차례예요. 바이브 코딩의 핵심은 말로 요청해서 원하는 결과를 얻는 것입니다. 코드를 직접 열 필요가 없어요.

프로젝트 설명 한 줄을 바꾸고 싶다면 이렇게 말하면 돼요.

"프로젝트 1의 설명을 'Pandas와 Folium으로 서울시 25개 구 카페 밀집도를 지도에 시각화했어요'로 바꿔주세요."

Claude는 `index.html` 안에서 해당 텍스트를 찾아 바꿔줍니다. 제목, 링크도 같은 방식으로 고칠 수 있어요.

내용을 고칠 때 한 번에 여러 개를 묶어서 요청해도 됩니다. "프로젝트 1 제목을 '카페 밀집도 지도'로, 링크를 `https://github.com/minji/cafe-map`으로 바꿔주세요"처럼 한 문장 안에 여러 변경사항을 담으면 Claude가 한꺼번에 처리해줘요. 하나씩 따로 요청해도 되지만, 관련 내용은 묶어서 말하는 게 더 효율적이에요.

프로젝트 순서를 바꾸고 싶을 때도 말로 해결할 수 있어요. 직접 HTML을 열어서 블록을 잘라 붙이는 대신, 이렇게 요청하면 됩니다.

"프로젝트 카드 순서를 바꿔주세요. 첫 번째를 '동아리 홈페이지', 두 번째를 '서울 카페 데이터 분석', 세 번째를 '뉴스 클론 코딩'으로 재배치해 주세요."

Claude가 세 `article` 블록의 위치를 바꿔줍니다. 이름과 원하는 순서만 말하면 되고, HTML 구조를 알 필요가 없어요. 순서 변경처럼 단순해 보이는 작업도 코드에서는 블록 단위로 잘라 붙여야 해서 실수가 생기기 쉬운데, 말로 요청하면 Claude가 정확하게 처리해줍니다.

수정 요청을 할 때 "어느 카드의 무엇을"을 구체적으로 말하는 것이 중요해요. "프로젝트 설명 바꿔주세요"보다 "프로젝트 1(서울 카페 데이터 분석)의 설명을 바꿔주세요"처럼 카드를 특정해야 Claude가 엉뚱한 카드를 건드리지 않아요. 특히 카드 내용이 비슷한 경우에는 고유한 키워드를 같이 언급하는 습관을 들이세요.

수정을 여러 번 반복하다 보면 Claude가 이전 요청의 맥락을 잃는 경우가 드물게 있어요. 응답이 이상하다면 "현재 index.html에서 프로젝트 카드 부분만 보여주세요"라고 먼저 확인 요청을 하고, 그 다음에 수정 요청을 이어가면 더 정확해요. 대화가 길어질수록 이런 확인 요청을 중간중간 넣는 게 도움이 됩니다.

수정 후에는 꼭 브라우저에서 결과를 확인하세요. Claude가 틀린 카드를 고쳤거나, 설명이 두 곳에 중복으로 들어간 경우가 드물게 있어요. 눈으로 확인하는 게 가장 빠른 검증이에요. 이상하다면 "방금 수정한 결과가 이상해요. 프로젝트 1 설명이 두 번 들어간 것 같아요. 확인해 주세요"처럼 구체적으로 말하면 됩니다.

> **TIP:** 프로젝트 이미지가 있으면 "프로젝트 1 카드에 `project1.png` 섬네일을 넣어주세요"처럼 말할 수 있어요. 이미지 추가는 Ch1에서 `me.jpg`를 넣은 방식과 동일해요. 파일을 폴더에 넣고 Claude에게 요청하면 됩니다. 이번 챕터에서는 텍스트만 다뤄요.

## Claude가 가끔 놓치는 것들

바이브 코딩으로 빠르게 결과를 얻을 수 있지만, Claude가 자주 빠뜨리는 세부 사항이 있어요. 직접 확인하고 말로 고치면 됩니다. 이 목록은 "Claude가 틀렸다"는 게 아니라 처음 요청에 명시하지 않았기 때문에 빠진 것들이에요. 아래 항목을 처음 프롬프트에 포함시키면 아예 발생하지 않고, 빠졌더라도 한 줄 요청으로 바로 해결돼요.

**`target="_blank"`는 있는데 `rel="noopener"`가 없는 경우.** 새 탭으로 여는 링크에는 보안상 `rel="noopener noreferrer"`를 함께 붙여야 해요. 없어도 동작은 하지만, 보안 취약점이 생길 수 있어요. 고치는 프롬프트: "카드 링크 전체에 `rel=\"noopener noreferrer\"`를 추가해 주세요."

**명함 CSS 선택자가 카드에도 영향을 주는 경우.** `.card p`처럼 넓은 선택자를 쓰면 나중에 추가한 `.project-card p`에도 스타일이 겹쳐 적용될 수 있어요. 카드 설명 글자 크기가 이상하게 크거나 색이 다르다면 이 충돌 때문일 가능성이 높아요. 고치는 프롬프트: "`.card` 안의 `p` 스타일이 `.project-card p`에도 적용되는 것 같아요. 선택자를 분리해서 서로 영향을 주지 않게 해주세요."

**링크 텍스트가 모든 카드에서 "자세히 보기"로 동일한 경우.** 스크린 리더 사용자나 탭 키로 탐색하는 사람에게는 구분이 안 돼요. 고치는 프롬프트: "각 카드 링크 텍스트를 '서울 카페 데이터 분석 보기', '뉴스 클론 코딩 보기'처럼 프로젝트 이름을 포함하게 바꿔주세요."

이런 세부 사항은 처음엔 눈에 잘 안 띄어요. 하지만 한 번 알아두면 말 한 마디로 바로 고칠 수 있어요.

바이브 코딩에서 Claude를 잘 쓴다는 건 처음 한 번에 완벽한 결과를 뽑아내는 게 아니에요. "이거 빠진 것 같아요, 추가해 주세요"를 자연스럽게 주고받는 과정 자체가 작업 흐름이에요. 처음 프롬프트로 80%를 얻고, 나머지 20%는 확인하면서 말로 채워나가는 감각을 익히는 게 이 챕터의 진짜 목표예요.

이 섹션에서 언급한 세 가지 함정—`rel="noopener"` 누락, 선택자 충돌, 링크 텍스트 중복—은 Claude만의 문제가 아니에요. 사람이 직접 HTML을 짤 때도 똑같이 빠뜨리는 것들이에요. 다만 말로 요청하는 방식에서는 "확인해줘"라는 한 마디로 Claude가 훑어봐 줄 수 있어요. "카드 링크에 보안 관련 속성이 올바르게 들어갔는지 확인하고, 빠진 게 있으면 추가해 주세요"처럼 검토 요청을 별도로 보내는 습관이 완성도를 높여줍니다.

## 최종 index.html 확인

Ch2가 끝났을 때 `index.html`은 아래처럼 되어 있어야 해요. 이 블록은 Ch2 완성본 전체를 담은 참조 스냅샷이에요. 직접 비교해보고, 크게 다르다면 Claude에게 "이 구조에 맞게 다시 정리해 주세요"라고 요청해도 됩니다.

코드 전체를 외울 필요는 없어요. 대신 구조를 눈으로 훑어보는 습관을 들여두면 좋아요. `<div class="card">`가 명함 블록이고, `<section class="projects">` 아래에 `<article class="project-card">`가 3개 있는 형태가 맞는지만 확인하면 됩니다. 이 구조가 흔들리면 말로 요청해도 엉뚱한 곳이 바뀔 수 있어요.

완성본 HTML에서 눈여겨볼 점이 하나 있어요. `<a>` 태그에 `target="_blank"`와 함께 `rel="noopener noreferrer"`가 붙어 있어요. 앞서 "Claude가 가끔 놓치는 것들" 섹션에서 언급한 보안 속성이에요. 이 완성본은 이미 반영되어 있으니, Claude가 만든 결과물에 이 속성이 빠져 있다면 완성본과 비교해서 추가 요청하면 됩니다. 링크 텍스트도 "자세히 보기" 대신 프로젝트 이름을 포함한 형태로 되어 있는 것도 확인해두세요.

## 여기까지 왔다면

명함 한 장에서 명함 + 카드 3장으로. 페이지에 내 이야기가 조금씩 쌓이고 있어요.

프롬프트 한 번으로 섹션이 통째로 생겼고, 카드 안에 내 프로젝트 정보가 들어갔어요. 코드를 한 줄도 직접 쓰지 않았지만, 결과물은 구체적이고 실제로 동작합니다. 이게 바이브 코딩의 감각이에요.

이 챕터에서 배운 패턴—"기존 것은 그대로 두고, 아래에만 추가해 주세요"—은 앞으로도 계속 쓰게 될 핵심 문장이에요. 무언가를 새로 붙일 때마다 Claude에게 "어디까지 건드려도 되는지"를 명확히 말해주면, Claude는 범위를 넘지 않아요. 반대로 말을 아끼면 Claude는 선의로 더 많이 바꿔버릴 수 있어요.

지금 이 시점에서 `index.html`을 한 번 따로 저장해두는 것도 좋은 습관이에요. `index-ch02-backup.html`처럼 이름을 바꿔서 같은 폴더에 복사해두면, Ch3 작업 중에 뭔가 잘못됐을 때 되돌아올 기준점이 생겨요. Claude가 실수로 구조를 크게 바꿨을 때 "Ch2 완성본으로 되돌려 주세요"라고 요청하는 것보다 백업 파일을 그대로 쓰는 게 훨씬 확실해요.

백업을 만들어두면 실험도 더 과감하게 할 수 있어요. "색을 완전히 바꿔봐 주세요"나 "카드 디자인을 다른 스타일로 바꿔봐 주세요" 같은 큰 변화를 시도하기가 훨씬 편해져요. 언제든 백업으로 돌아올 수 있다는 안전망이 있으면 Claude와 주고받는 대화가 더 자유로워집니다.

다음 챕터에서는 이 구조를 그대로 두고, 색·폰트·여백을 내 취향으로 바꿔서 전혀 다른 인상으로 만들어볼게요. 레이아웃은 손대지 않고 스타일만 바꾸는 작업인데, 프롬프트 한 줄의 차이가 페이지 전체 분위기를 뒤집는 경험을 하게 될 거예요.

## 현재 프로젝트 상태

```
my-portfolio/
  ├─ index.html
  └─ me.jpg
```

파일 개수는 Ch1과 동일해요. `index.html` 하나가 커졌을 뿐입니다. 명함 코드 위에 프로젝트 섹션이 쌓인 거예요.

Ch3에서 스타일을 바꾸고, Ch4에서 반응형을 추가하고, Ch5에서 배포해도 이 폴더 구조는 그대로예요. 파일이 늘어나는 시점은 이미지를 추가하거나 별도 CSS 파일을 분리할 때인데, 그건 이 책의 범위 밖이에요. 지금은 단순한 구조를 유지하면서 내용을 탄탄히 채우는 게 먼저입니다.

파일이 늘어나지 않았다는 점이 중요해요. 별도 `projects.html`을 만들거나 JavaScript 파일을 추가하지 않았어요. HTML 한 파일 안에서 섹션이 늘어나는 방식이 바이브 코딩 초반 단계의 기본 패턴이에요. 구조가 단순할수록 Claude와 주고받는 대화도 단순해지고, 실수가 생겼을 때 되돌리기도 쉬워요. Ch4에서 반응형 레이아웃을 추가할 때도, Ch5에서 배포할 때도 이 단일 파일 구조가 그대로 유지돼요.

## 이번 챕터 체크리스트

- [ ] 내 프로젝트 3개를 제목·설명·링크로 정리했다
- [ ] 프롬프트에 `target="_blank"`와 "명함은 그대로 두고" 표현을 넣었다
- [ ] Claude가 `index.html`에 추가했는지 (새 파일이 생기지 않았는지) 확인했다
- [ ] 브라우저에서 명함 아래에 카드 3장 그리드가 표시된다
- [ ] 카드 링크를 클릭하면 새 탭이 열린다
- [ ] body 레이아웃이 수직 스택으로 재조정되어 명함과 카드 섹션이 자연스럽게 이어진다
