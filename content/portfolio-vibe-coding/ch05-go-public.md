# 세상에 공개: GitHub + Vercel

Ch4에서 내 포트폴리오가 화면 크기와 시간대에 맞춰 스스로 적응하게 됐어요. 이제 이 `index.html`을 세상에 꺼내서 **누구나 접속할 수 있는 링크** 하나로 만들어볼게요.

---

## 이번 챕터의 목표

- GitHub에 내 포트폴리오 코드를 저장할 레포(repository) 만들기
- 로컬 `index.html`과 `me.jpg`를 GitHub에 올리기 — Claude가 대행
- Vercel 대시보드에서 "Import Git Repository"로 배포 연결하기
- `https://내이름.vercel.app` 형태의 공개 URL을 받아서 이력서에 붙이기

---

## 배포 전에 확인할 것

이 챕터를 시작하기 전에 딱 하나만 확인해요. `my-portfolio` 폴더 안에 `index.html`과 `me.jpg` 두 파일이 있는지예요. Ch4를 마쳤다면 이미 있어요. 파일이 없거나 이름이 다르면 Claude에게 "현재 폴더에 어떤 파일이 있는지 확인해줘"라고 물어봐요.

GitHub 계정과 Vercel 계정도 이미 있어야 해요. Ch0에서 만들었어요. 로그인 상태인지 각 사이트에서 미리 확인해두면 이 챕터 진행이 더 매끄러워요. 계정 정보가 기억이 안 난다면 지금 각 사이트에 접속해서 로그인 여부를 확인해봐요.

두 가지 모두 준비됐다면 1단계부터 순서대로 진행해요. 중간에 막히는 구간이 생기면 Claude에게 에러 메시지나 화면 설명을 그대로 전달하면 돼요. 이 챕터 어디서든 막히는 지점이 있을 때 "부록 A"에 주요 케이스가 정리돼 있으니 참고해요.

---

## 왜 두 서비스가 필요한가요?

GitHub와 Vercel, 두 서비스를 쓰는 게 처음엔 번거롭게 느껴질 수 있어요. "그냥 한 곳에 올리면 안 되나요?" 하는 생각이 드는 게 당연해요.

역할이 달라서 그래요. **GitHub**는 코드 저장소예요. 내 `index.html`이 언제 어떻게 바뀌었는지 기록을 남기고, 그 코드가 내 계정에 공개적으로 남아요. 이력서에 GitHub 링크를 넣으면 채용 담당자가 "이 사람이 실제로 코드를 관리했구나"라는 인상을 받아요. **Vercel**은 그 코드를 읽어서 브라우저에서 접근 가능한 URL로 배포해주는 서비스예요. 누군가 그 URL을 치면 내 포트폴리오가 바로 떠요.

비유하자면 GitHub는 "원고 파일을 저장해두는 드라이브"이고, Vercel은 "그 원고를 누구나 읽을 수 있게 출판해주는 곳"이에요. 드라이브에만 있으면 본인만 볼 수 있고, 출판이 되어야 세상이 볼 수 있어요. 두 역할이 분리돼 있기 때문에 각자가 자기 일을 잘 할 수 있어요.

이 조합을 쓰는 이유가 몇 가지 더 있어요. 둘 다 **무료**고, **클릭 중심 UI**로 작동해요. 터미널 명령어나 서버 설정 없이 웹 브라우저에서 버튼 클릭만으로 배포까지 끝낼 수 있어요. 그리고 내 코드가 GitHub에 남으니까, Vercel 공개 URL과 GitHub 레포 링크를 이력서에 동시에 첨부할 수 있어요. 공개 URL 하나가 아니라, 코드를 직접 보여줄 수 있는 레포도 덤으로 생기는 거예요.

또 한 가지 중요한 점이 있어요. Vercel은 GitHub에 새 코드가 올라올 때마다 **자동으로 재배포**를 해줘요. 한 번 연결해두면 앞으로 포트폴리오를 수정할 때마다 따로 배포 버튼을 누를 필요가 없어요. Claude에게 수정을 요청하고 GitHub에 올리는 것만으로 공개 URL이 자동으로 최신 상태가 돼요. 이 흐름이 익숙해지면 포트폴리오를 "한 번 만들고 끝"이 아니라 "계속 다듬어 가는 것"으로 쓸 수 있어요.

실무에서도 개발자들이 똑같은 방식으로 일해요. 코드를 고치고, GitHub에 올리고, 배포가 자동으로 되는 흐름이요. 이번 챕터에서 그 흐름을 처음으로 직접 경험하게 돼요. 규모의 차이는 있지만 구조는 동일해요.

이 챕터에서 하는 일을 순서대로 그려보면 이래요.

1. github.com에서 레포를 만들어요 → 코드가 올라갈 "빈 방"이 생겨요
2. Claude가 로컬 파일을 그 방에 올려줘요 (`git push`) → 코드가 GitHub에 저장돼요
3. vercel.com에서 그 레포를 가져와요 (Import) → 코드가 공개 URL로 배포돼요
4. 이후에는 로컬 수정 → `git push` → Vercel 자동 재배포 → URL 업데이트

이 흐름이 이 챕터 전체예요.

한 가지 더 짚어두면, 이 방식은 "한 번 설정하면 자동"이에요. 3단계까지가 초기 설정이고, 4단계부터는 로컬에서 파일을 고치고 Claude에게 "올려줘"라고 말하는 것만으로 공개 URL이 자동으로 최신 상태가 돼요. 포트폴리오를 계속 업데이트하는 루틴이 생기는 게, 결국 이 챕터에서 얻는 가장 큰 자산이에요.

> **TIP** Ch0에서 Vercel 가입 시 'Continue with GitHub'을 눌렀다면, 이 챕터가 한결 빨라요. 계정이 이미 연결돼 있으니까요. 기억이 나지 않는다면 이 챕터를 진행하다가 Vercel이 GitHub 권한을 요청할 때 허용해주면 돼요.

## 1단계 — GitHub 레포 만들기

github.com에 로그인한 뒤 오른쪽 위 **`+`** 버튼을 클릭하면 드롭다운이 떠요. 거기서 **`New repository`**를 선택해요.

**레포 이름**을 입력해요. `my-portfolio`처럼 영문 소문자와 하이픈으로 만드는 게 깔끔해요. 레포 이름은 공개 URL에 직접 반영되지 않아요. Vercel이 따로 프로젝트 이름을 물어봐요. 하지만 GitHub에서 코드 저장소로 보이는 이름이니까, 나중에 이력서에 GitHub 링크를 붙일 때 알아보기 쉽게 정해두는 게 좋아요.

**공개 범위**는 반드시 **Public**으로 선택해요. 이력서에 올릴 공개 포트폴리오니까 Public이 자연스러워요.

**README, `.gitignore`, License**는 모두 체크하지 않아요. **빈 레포**로 시작해야 로컬 파일을 올릴 때 충돌이 없어요. 파일이 하나라도 있으면 다음 단계에서 `git push`할 때 "이미 커밋이 있다"는 충돌 에러가 나요. 이게 초보자가 가장 자주 만나는 함정이에요. 빈 레포로 시작해서 로컬 파일을 그대로 밀어 올리는 게 제일 깔끔해요.

**`Create repository`** 버튼을 누르면 레포가 만들어져요. 생성 직후 "Quick setup" 화면이 보여요. 이 화면은 "아직 아무 파일도 없는 빈 레포"를 로컬과 어떻게 연결할지 안내하는 거예요. 여기서 **HTTPS URL**을 복사해 메모해 둬요. 형태는 이런 식이에요.

```
https://github.com/minji/my-portfolio.git
```

이 URL이 다음 단계에서 Claude에게 전달할 레포 주소예요. 창을 닫기 전에 꼭 복사해두세요. 다음 단계에서 Claude가 이 주소를 받아 나머지 연결 작업을 전부 처리해줘요.

레포 이름에 대해 한 가지 더 짚어둘게요. `my-portfolio`가 이미 다른 계정에 있어도 상관없어요. GitHub 레포 이름은 "내 계정 안에서 중복되지 않으면" 되기 때문에, `github.com/minji/my-portfolio`와 `github.com/junho/my-portfolio`는 완전히 다른 레포예요. 이름이 겹쳐도 계정이 다르면 아무 문제 없어요.

레포를 만들고 나면 잠시 멈춰서 이 상황을 실감해봐요. 내 이름이 붙은 공개 공간이 인터넷에 생긴 거예요. 아직 빈 방이지만, 곧 Ch1부터 지금까지 쌓아온 코드가 거기 들어갈 거예요.

<!-- IMAGE: screenshot-ch05-github-new-repo.png | GitHub 레포 생성 폼. Repository name에 'my-portfolio' 입력, Public 선택, README/gitignore/License 세 항목 모두 미체크 상태. Create repository 버튼 직전 화면. -->

> **WARNING** Public 대신 Private을 선택하면 Vercel이 레포에 연결할 때 권한 문제가 나올 수 있어요. 이력서에 올릴 공개 포트폴리오니까 Public이 자연스럽고, 나중에 Settings에서 언제든 변경할 수 있어요.

## 2단계 — Claude에게 GitHub 업로드 대행 요청

1단계가 끝나면 GitHub에 빈 레포가 만들어진 상태예요. 이제 로컬 `my-portfolio` 폴더 안에 있는 파일을 그 레포에 올릴 차례예요. 전통적으로는 터미널에서 git 명령어를 직접 치는 과정인데, 이 책에서는 Claude에게 말로 요청해서 대행시키는 방식을 써요. 터미널 명령어를 외울 필요가 없고, 실수할 여지도 줄어요.

`my-portfolio` 폴더 안에서 `claude`를 실행한 상태에서 Claude에게 이렇게 말해요.

```markdown
이 프로젝트를 제 GitHub 레포에 올려주세요.
레포 주소: https://github.com/minji/my-portfolio.git
커밋 메시지는 "첫 포트폴리오 공개"로 해주세요.
```

`minji` 자리에 본인 GitHub 사용자 이름을, `my-portfolio`는 방금 만든 레포 이름으로 바꿔요. 레포 주소는 앞 단계에서 복사해 둔 HTTPS URL이에요.

Claude가 내부적으로 실행하는 명령어는 아래와 같아요. 이 블록은 참고용이에요. 여러분이 직접 칠 필요는 없어요.

```bash
git init
git add index.html me.jpg
git commit -m "첫 포트폴리오 공개"
git branch -M main
git remote add origin https://github.com/minji/my-portfolio.git
git push -u origin main
```

각 명령어가 무엇을 하는지 짧게 짚어볼게요. `git init`은 현재 폴더를 "git이 추적하는 폴더"로 만드는 초기화 명령어예요. `git add`는 어떤 파일을 묶을지 선택해요. `git commit`은 묶인 파일을 "이 시점의 스냅샷"으로 저장해요. `git branch -M main`은 기본 브랜치 이름을 main으로 설정하고, `git remote add origin`은 "이 로컬 폴더를 저 GitHub 레포와 연결"해요. `git push -u origin main`이 드디어 실제로 GitHub에 올려요.

이 흐름 전체를 Claude가 알아서 실행해요. 잠깐 기다리면 "Successfully pushed" 또는 "push 완료" 같은 메시지가 나와요. 그러면 GitHub 레포 페이지(`github.com/내이름/my-portfolio`)를 새로고침해서 `index.html`과 `me.jpg`가 올라왔는지 눈으로 확인해요.

파일 두 개가 목록에 보이고, 커밋 메시지 "첫 포트폴리오 공개"가 옆에 표시되면 이 단계는 끝이에요. 방금 생성한 GitHub 레포가 더 이상 빈 방이 아니에요. 이제 코드가 들어 있어요.

`git push`가 처음으로 성공하는 순간이 꽤 인상적이에요. 내 컴퓨터에만 있던 파일이 인터넷 어딘가에 올라갔다는 게 실감나거든요. GitHub 레포 페이지에서 `index.html` 파일 이름을 클릭하면 코드 내용도 직접 볼 수 있어요. Ch1부터 지금까지 쌓인 코드 전체가 거기 있어요.

한 가지 알아두면 좋은 게 있어요. `git commit` 메시지는 나중에 히스토리를 볼 때 "이 시점에 무슨 변경이 있었는지"를 알려주는 메모예요. "첫 포트폴리오 공개"는 첫 배포에 딱 어울리는 메시지예요. 앞으로 포트폴리오를 수정할 때마다 "프로젝트 3 업데이트", "연락처 추가" 같은 식으로 무엇을 바꿨는지 남겨두면, 나중에 히스토리를 돌아볼 때 한눈에 파악돼요. Claude에게 커밋 메시지를 함께 알려주면 그대로 써줘요.

<!-- IMAGE: screenshot-ch05-github-pushed.png | GitHub 레포에 index.html과 me.jpg가 올라간 화면. 파일 목록과 커밋 메시지 "첫 포트폴리오 공개"가 보이는 상태. -->

> **NOTE** 처음 `git push` 시 GitHub이 **Personal Access Token** 또는 GitHub Desktop 로그인을 요구할 수 있어요. 터미널에서 직접 로그인하거나, GitHub Desktop을 설치해서 연결하거나, Token을 한 번 만들어 저장하면 다음부터는 묻지 않아요. Claude에게 '이 에러 어떻게 해결해?' 라고 에러 메시지와 함께 물어보면 상황에 맞는 안내를 바로 해줘요.

> **WARNING** 인증 오류나 `remote: Repository not found` 같은 메시지가 뜨면 **부록 A의 'Vercel Import 실패 / GitHub 인증 오류' 케이스**를 참고해요. 에러 메시지 전체를 Claude에게 그대로 붙여넣어도 바로 진단해줘요.

## 3단계 — Vercel Import UI로 배포

GitHub에 코드가 올라갔으니 이제 Vercel에서 그 레포를 가져올 차례예요. 이 단계가 끝나면 공개 URL이 생겨요. Vercel 대시보드에서 클릭 몇 번으로 끝나고, 터미널을 쓸 필요가 없어요.

vercel.com에 로그인해요. 대시보드 오른쪽 위 **Add New → Project**를 눌러요.

**"Import Git Repository"** 섹션이 보여요. Ch0에서 Vercel을 GitHub 계정으로 가입했다면 내 GitHub 레포 목록이 자동으로 나타나요. 방금 만든 `my-portfolio`를 찾아서 옆의 **`Import`** 버튼을 눌러요.

프로젝트 설정 화면이 열려요. 프로젝트 이름은 기본값 그대로 둬도 돼요. 여기서 한 가지만 꼭 바꿔요. **Framework Preset** 드롭다운을 찾아서 **`Other`**를 선택해요. 우리 포트폴리오는 순수 HTML 파일이라 React나 Next.js 같은 빌드 도구가 필요 없어요. "Other"로 설정하면 Vercel이 `index.html`을 그대로 서빙해요.

나머지 설정(Root Directory, Build Command, Output Directory)은 모두 기본값 그대로 두고 **`Deploy`** 버튼을 눌러요. 30~60초 정도 진행 바가 움직이면서 빌드가 돌아요.

이 과정에서 Vercel이 하는 일은 간단해요. GitHub 레포를 자기 서버로 복사해서 `index.html`을 외부에서 접근 가능한 주소에 연결해요. "빌드"라고 하지만 우리 프로젝트에는 빌드할 코드가 없어요. 파일을 그대로 올리는 게 전부예요. 그래서 배포가 30~60초 안에 끝나요.

이 시점에서 Vercel은 GitHub 레포의 `main` 브랜치를 감시하기 시작해요. 앞으로 `main` 브랜치에 새 커밋이 올라오면 Vercel이 자동으로 감지해서 재배포를 시작해요. 수동으로 배포 버튼을 누를 필요가 없는 이유가 이거예요.

배포가 진행되는 동안 Vercel 화면에는 로그가 실시간으로 흘러요. "Cloning repository", "Building" 같은 메시지가 차례로 나와요. 전부 녹색이면 성공이에요. 중간에 빨간 글씨가 보이면 뭔가 문제가 생긴 거예요. 그 에러 메시지를 복사해서 Claude에게 보여주면 바로 원인을 찾아줘요.

<!-- IMAGE: screenshot-ch05-vercel-import.png | Vercel Import 화면. my-portfolio 레포를 선택하고 Framework Preset 드롭다운에서 'Other'를 선택한 상태. Deploy 버튼이 활성화된 화면. -->

> **TIP** Framework Preset을 **Other**로 두면 Vercel이 `index.html`을 자동으로 루트로 서빙해요. Next.js 또는 Create React App을 잘못 선택하면 "빌드 명령어를 찾을 수 없다"는 에러가 나요. 우리 포트폴리오에는 "Other"가 정답이에요.

Ch0에서 Vercel을 GitHub 연동으로 가입하지 않았다면 레포 목록이 비어 있을 수 있어요. 그때는 화면 아래 "Adjust GitHub App Permissions" 링크를 눌러서 GitHub에 Vercel 접근 권한을 한 번만 허용해주면 돼요. 허용하고 나면 레포 목록이 바로 나타나요.

## 4단계 — 공개 URL 확인

빌드가 완료되면 Vercel이 축하 화면을 보여줘요. 화면 가운데 공개 URL이 크게 표시돼요. 형태는 이런 식이에요.

```
https://my-portfolio-xxx.vercel.app
```

뒤에 붙은 해시값(xxx 부분)은 Vercel이 자동으로 생성해요. 그 URL을 클릭하면 내 포트폴리오가 열려요. 지금까지 로컬 파일(`file:///...`)로만 보던 그 화면이, 이제는 인터넷 어디서든 접속되는 주소에 살아 있어요.

이 순간이 이 책 전체에서 가장 실감나는 순간이에요. 지금 당장 핸드폰을 꺼내서 그 주소를 직접 쳐봐요. Ch4에서 만든 반응형 레이아웃이 핸드폰에서도 깔끔하게 보이는지, 다크모드 토글 버튼도 눌러봐요. 내 손으로 만든 포트폴리오가 다른 기기에서도 그대로 보인다는 게, 직접 경험해야 비로소 크게 느껴져요.

공개 URL을 복사해서 메모장이나 노션에 저장해 두세요. 다음 단계에서 이력서에 넣을 때 바로 쓸 수 있도록요. 그리고 지인에게 한 명만 링크를 공유해봐요. "내가 포트폴리오 만들었어, 한번 봐봐"라고요. 혼자 로컬에서 보던 것과 다른 사람이 실제로 접속해서 "오 진짜 떴다"고 반응하는 게 느낌이 완전히 달라요.

배포 완료 화면에서 Vercel은 URL을 여러 형태로 보여줘요. 자동 생성된 주소 말고, 나중에 Settings → Domains에서 짧은 이름으로 바꿀 수 있어요. 이력서에 넣기 전에 짧은 주소로 바꾸는 걸 추천해요.

URL을 짧게 바꾸는 방법은 간단해요. Vercel 대시보드에서 프로젝트를 클릭한 뒤 **Settings → Domains** 탭으로 가요. 여기서 원하는 이름을 입력하고 추가하면 돼요. `my-portfolio.vercel.app`이 사용 가능하다면 바로 적용돼요. 이미 누군가가 쓰고 있다면 빨간 표시가 나와요. 그때는 `minji-portfolio.vercel.app`처럼 이니셜이나 본인만의 단어를 앞에 붙여서 시도해봐요. 한 번 확정되면 그 주소가 영구적으로 내 포트폴리오 URL이 돼요.

확인할 항목을 체크해봐요.

- [ ] 공개 URL을 브라우저에서 열었을 때 포트폴리오 화면이 제대로 뜨는지
- [ ] 핸드폰에서 같은 URL로 접속했을 때 모바일 레이아웃이 적용되는지
- [ ] 다크모드 토글을 눌렀다가 새로고침해도 설정이 유지되는지

<!-- IMAGE: screenshot-ch05-vercel-live.png | Vercel 대시보드의 Deployment 완료 화면. 초록색 체크 아이콘과 함께 공개 URL(my-portfolio-xxx.vercel.app)이 크게 표시된 상태. -->

> **NOTE** Vercel이 처음 자동으로 주는 URL은 `my-portfolio-<해시>.vercel.app`이에요. Settings → Domains에서 **`my-portfolio.vercel.app`** 같은 짧은 주소로 바꿀 수 있어요. 이미 다른 사람이 쓰고 있는 이름이면 안 되니까, 이름 앞에 이니셜이나 숫자를 붙여서 시도해봐요.

> **WARNING** 빌드가 실패(Deployment Failed)하면 대부분 파일 이름 문제예요. `Index.html`처럼 대문자로 저장됐거나, 한글 파일명이거나, 경로 오타가 있을 때 이런 일이 일어나요. **부록 A의 'Vercel Import 실패' 케이스**를 참고해서 Claude에게 에러 스크린샷을 보여주면 원인과 해결 방법을 바로 알려줘요.

## 5단계 — 이력서에 URL 붙이기

공개 URL이 생겼으니 이걸 이력서에 넣어야 진짜 완성이에요. 여기까지 오는 데 걸린 시간이 얼마든, 이 URL 한 줄이 이력서에 들어가는 순간 포트폴리오가 실질적인 역할을 하기 시작해요.

Notion 이력서든, PDF든, 한글 문서든 "개인 프로젝트" 또는 "자기소개" 영역에 URL을 넣어요. 바로 붙여 쓸 수 있는 예시 문장 두 가지예요.

> "개인 포트폴리오: https://my-portfolio.vercel.app — 데이터로 세상을 더 쉽게 만드는 사람을 한 페이지에 담았어요."

> "프로젝트 3개와 자기소개를 반응형 + 다크모드로 구현했어요(https://my-portfolio.vercel.app)."

문장은 본인 상황에 맞게 바꿔도 돼요. 중요한 건 URL이 **클릭 가능한 하이퍼링크** 형태로 들어가야 한다는 거예요. PDF나 한글 문서에서는 URL을 입력하면 자동으로 링크가 걸리는 경우가 많아요. Notion에서는 텍스트를 선택하고 `Cmd/Ctrl + K`로 링크를 거세요.

bit.ly 같은 URL 단축기는 쓰지 않는 게 좋아요. 단축 URL은 원본 링크가 뭔지 알 수 없어서 신뢰도가 낮아 보여요. `vercel.app` 도메인 자체가 이미 짧고 명확해요.

이력서에 넣을 때 GitHub 레포 주소(`github.com/내이름/my-portfolio`)도 함께 넣으면 더 좋아요. 채용 담당자가 코드를 직접 보고 싶을 때 클릭할 수 있는 링크가 생기는 거예요. Vercel 링크는 "완성된 결과물"을, GitHub 링크는 "과정이 담긴 코드"를 보여줘요. 두 링크를 나란히 쓰는 방식이 점점 일반적인 이력서 패턴이 되고 있어요.

핸드폰에 QR 코드를 만들어두는 방법도 있어요. 면접 자리에서 "QR 찍어보시면 바로 보실 수 있어요" 한 마디면 인상이 달라져요. QR 코드 생성 사이트에 URL을 넣으면 이미지로 다운받을 수 있어요. 명함에 인쇄하거나 발표 슬라이드에 넣는 방법으로도 활용해요.

URL을 이력서에 넣기 전에 한 번 더 확인해봐요. 복사한 주소를 시크릿 탭에서 열어서 로그인 없이도 포트폴리오가 잘 뜨는지 확인하는 거예요. 본인 계정에 로그인된 상태에서는 권한 문제가 숨겨질 수 있거든요. 시크릿 탭에서 열린다면 채용 담당자 화면에서도 똑같이 잘 열려요.

> **TIP** 면접에서 '포트폴리오 보여주세요' 할 때 이 URL을 바로 열어요. 다크모드 토글을 눌러 보여주면 이야깃거리 하나 더 생겨요. "반응형으로 만들어서 모바일에서도 같은 화면이에요"라고 한 마디 덧붙이면 기술 이해도가 자연스럽게 드러나요.

## 배포 후 한 번 더 점검할 것들

공개 URL이 생겼다고 바로 이력서에 붙이기 전에, 체크할 게 몇 가지 있어요. 세 가지만 확인하면 충분해요.

첫째, **파일 경로 확인**이에요. `me.jpg`가 포트폴리오에서 깨져 보이는 경우가 가끔 있어요. 로컬에서는 잘 보이던 사진이 배포 후에 안 보인다면, 파일 이름 대소문자 문제일 가능성이 높아요. 로컬 맥OS는 대소문자를 구분 안 하는 경우가 있지만, Vercel 서버는 구분해요. `Me.jpg`와 `me.jpg`가 다르게 인식돼요. Claude에게 "배포 후 이미지가 안 보여요"라고 하면 바로 원인을 찾아줘요.

둘째, **Google Fonts 로딩 확인**이에요. 로컬에서는 오프라인 캐시 덕분에 폰트가 잘 보이다가, 배포 환경에서 갑자기 기본 폰트로 뜨는 경우가 있어요. `index.html`의 `<link>` 태그에 Google Fonts 주소가 제대로 들어가 있는지 확인해봐요. Claude에게 "구글 폰트 링크 방식 확인해줘"라고 하면 현재 파일을 보고 맞는지 알려줘요.

셋째, **다크모드 초기 상태 확인**이에요. 처음 포트폴리오에 접속하는 사람은 `localStorage`에 이전 설정이 없어서 항상 라이트모드로 시작해요. 이게 기대한 동작인지 확인해봐요. 라이트모드 초기 상태가 이력서를 보는 사람에게 어떻게 보일지 직접 시크릿 탭에서 열어 확인해봐요.

이 세 가지가 모두 문제없으면 이력서에 URL을 붙여도 안심이에요.

## 고치고 싶을 때

배포가 끝났다고 끝이 아니에요. 포트폴리오는 한 번 올리고 굳히는 문서가 아니에요. 새 프로젝트가 생기면 카드를 추가하고, 자기소개 문장을 다듬고 싶을 때 열어서 고치면 돼요. 오히려 주기적으로 업데이트하는 포트폴리오가 "관리되고 있는 포트폴리오"라는 인상을 줘요.

수정 흐름은 처음 배포할 때보다 훨씬 간단해요. 로컬에서 `claude`를 열고 내용을 바꾼 뒤, Claude에게 이렇게 요청해요.

```markdown
프로젝트 2 설명을 '실시간 대시보드 구현, 누적 사용자 1,200명'으로 바꾸고,
바뀐 내용을 GitHub에 다시 올려주세요.
```

Claude가 `index.html`을 고치고, `git commit`과 `git push`를 실행해요. GitHub에 코드가 올라가면 Vercel이 자동으로 감지해서 30~60초 뒤에 재배포를 완료해요. 공개 URL을 새로고침하면 바뀐 내용이 바로 보여요. Vercel 대시보드에서 따로 배포 버튼을 누를 필요가 없어요.

이 자동화가 이 조합의 진짜 장점이에요. GitHub에 `git push`만 되면 Vercel이 알아서 새 버전을 배포해요. Claude에게 수정 요청 한 번이면 공개 URL이 업데이트돼요.

수정이 쌓이면 GitHub 레포의 커밋 히스토리에 기록이 남아요. "처음 공개 → 프로젝트 설명 보강 → 연락처 추가"처럼 포트폴리오가 변해온 과정이 코드 단위로 남는 거예요. 이게 단순한 파일 저장과 다른 점이에요. 나중에 "언제 어떻게 바꿨는지" 돌아볼 수 있어요.

포트폴리오를 고치는 게 번거롭다고 느끼면 결국 오래된 내용이 그대로 남아요. "Claude에게 말하면 고쳐줘요" + "GitHub에 올리면 배포돼요"가 자연스러운 습관이 되면, 포트폴리오가 계속 살아 있는 문서로 작동해요. 새 프로젝트를 마칠 때마다 포트폴리오를 열어서 카드 하나를 추가하는 게 루틴이 되면, 취업 준비 기간이 됐을 때 "지금 급하게 만들어야 한다"는 압박이 줄어요. 항상 최신 상태인 포트폴리오를 링크 하나로 보낼 수 있는 상태가 유지되니까요.

수정할 때 한 가지 팁을 드릴게요. Claude에게 "GitHub에 올려줘"라고 할 때 어떤 걸 바꿨는지 한 줄로 알려주면 커밋 메시지도 함께 잘 정해줘요. "프로젝트 2 설명 업데이트 — 이 내용으로 GitHub에 올려줘"처럼요. 커밋 메시지가 쌓이면 GitHub 레포 페이지에서 포트폴리오 변천사를 한눈에 볼 수 있어요. 작은 습관 하나가 나중에 이력서에 "꾸준히 관리해온 프로젝트"라는 인상을 남겨줘요.

혹시 수정한 내용이 공개 URL에 반영이 안 된다면 Vercel 대시보드의 Deployments 탭을 확인해봐요. 최신 배포 상태가 "Ready"면 정상이에요. "Error"면 에러 로그를 클릭해서 내용을 확인하고 Claude에게 보여주면 바로 해결해줘요. **부록 A의 'Vercel Import 실패' 케이스**에도 자주 나오는 패턴들이 정리돼 있어요.

## 여기까지 왔다면

**Ch0의 빈 폴더**에서 시작한 이야기가 **공개 URL**로 끝났어요.

지금 핸드폰으로 그 주소를 쳐 보면, 내가 만든 포트폴리오가 거기 있어요. 이력서에 붙일 URL이 하나 생겼고, 그 URL 뒤에는 내가 Claude와 대화하며 만든 코드가 있어요. Ch1에서 이름 하나 넣었던 명함 페이지가, Ch2에서 프로젝트 카드를 달고, Ch3에서 내 취향으로 색이 입혀지고, Ch4에서 환경에 맞춰 적응하게 되더니, 이제는 세상 어디서든 접속 가능한 주소로 살아 있어요.

코드를 몰랐던 사람이 Claude와 대화만으로 여기까지 왔다는 게, 직접 겪어보면 생각보다 훨씬 크게 느껴져요. 세상은 이제 이 포트폴리오를 볼 수 있어요.

잠깐 멈추고 지금 이 상황을 실감해봐요. 이 포트폴리오에 접속하는 사람은 누군가의 노트북이거나, 다른 나라의 스마트폰이거나, 면접관의 태블릿일 수도 있어요. Ch4에서 만든 반응형 덕분에 어떤 화면에서도 깔끔하게 열리고, 다크모드 덕분에 어떤 환경에서도 편하게 읽혀요. 코드를 직접 짜지 않았어도, 설계는 내가 했고, 선택은 내가 했고, 완성은 내가 해냈어요.

앞으로 포트폴리오는 계속 달라질 거예요. 프로젝트가 늘어나고, 자기소개가 다듬어지고, 어쩌면 완전히 다른 디자인으로 바뀔 수도 있어요. 그때마다 Claude에게 요청하고, GitHub에 올리고, Vercel이 자동으로 배포해줘요. 한 번 만든 파이프라인이 계속 살아서 작동해요. 이게 이번 챕터에서 진짜로 얻은 것이에요. 포트폴리오 한 페이지가 아니라, 앞으로 계속 쓸 수 있는 배포 루틴이요.

## 다음에 더 하고 싶다면

이 책의 본편은 여기서 끝이에요. 하지만 포트폴리오는 여기서 멈추지 않아도 돼요. 더 나아가고 싶은 방향이 생기면 아래를 참고해요.

**커스텀 도메인**을 붙이고 싶다면 부록 C를 참고해요. `https://minji.dev` 같은 내 이름이 들어간 주소로 바꾸는 방법이 나와 있어요. 도메인 구매부터 Vercel 연결까지 순서대로 안내해요.

**Claude Code 이외의 방식**으로 git을 다루고 싶다면 부록 C의 대안 경로를 참고해요. GitHub Desktop이나 다른 도구로도 같은 결과를 낼 수 있어요.

**포트폴리오 템플릿**을 새로 시작하거나 구조를 크게 바꾸고 싶다면 부록 B의 템플릿 4종을 참고해요. 명함형, 카드형, 감성형 등 다양한 출발점이 준비돼 있어요.

## 이번 챕터 체크리스트

모든 항목이 체크되면 포트폴리오 바이브 코딩의 본편이 끝난 거예요. 아직 안 된 항목이 있으면 해당 섹션으로 돌아가서 Claude에게 다시 요청해봐요. 체크리스트는 "완료 확인"이 아니라 "빠진 게 없는지 점검"하는 도구예요.

중간에 막히는 항목이 있으면 부록 A의 케이스 목록을 먼저 살펴보고, 해당하는 상황이 없다면 Claude에게 증상을 그대로 설명하면 돼요. 막힘 없이 끝까지 온다면, 체크리스트를 다 채운 순간이 이 책의 마지막 페이지예요.

- [ ] GitHub 레포를 Public으로 생성했어요
- [ ] Claude 대행으로 `git push` 완료했어요
- [ ] GitHub 레포에 `index.html`과 `me.jpg`가 올라온 걸 확인했어요
- [ ] Vercel Import에서 Framework Preset을 **Other**로 설정하고 배포했어요
- [ ] `https://내이름.vercel.app` 형태의 공개 URL이 발급됐어요
- [ ] 핸드폰에서 그 URL로 접속해서 반응형 + 다크모드 토글이 작동하는 걸 확인했어요
- [ ] 시크릿 탭에서 공개 URL을 열어 로그인 없이도 포트폴리오가 잘 뜨는지 확인했어요
- [ ] 이력서 "개인 프로젝트" 또는 "자기소개" 영역에 URL을 하이퍼링크로 넣었어요
