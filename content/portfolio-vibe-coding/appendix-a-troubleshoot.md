# 부록 A: 트러블슈팅 가이드

포트폴리오를 만들다 보면 생각하지 못한 데서 막힐 수 있어요. 대부분은 드물지 않은 실수고, 대화 한두 번이면 풀려요. Claude에게 상황을 설명하면 거의 모든 문제가 해결되는데, 어떻게 설명해야 할지 막막할 때 이 부록이 도움이 될 거예요.

이 부록은 자주 나오는 6가지 케이스를 모아둔 곳이에요. 막히는 순간 빠르게 찾아볼 수 있게 각 케이스를 같은 구조로 정리했어요: **증상 → 원인 → 해결 프롬프트 → 확인**. Ch2, Ch5 같은 챕터에서 WARNING 콜아웃이 '부록 A 참조'로 보낸 경우가 있어요. 거기서 왔다면 해당 번호 케이스로 바로 내려가세요.

에러는 무섭지 않아요. 어떤 에러가 나왔는지 그 자체가 이미 원인의 절반이에요. 빨간 글씨를 보고 당황하지 말고, 그 내용을 그대로 Claude에게 보내는 게 가장 빠른 해결이에요. 이 부록의 케이스와 맞아떨어지지 않는 상황이더라도, 마지막 "추가 도움이 필요하다면" 섹션에 범용 대처법이 있어요.

아래 표에서 자신의 상황과 가장 가까운 케이스를 먼저 찾아보세요. 각 케이스는 독립적으로 읽을 수 있어요. 순서대로 읽을 필요 없이 해당 번호로 바로 이동하면 돼요.

|--------|----------------|----------------|
| 케이스 1 | Claude가 index.html 대신 새 파일을 만들었어요 | Ch2 |
| 케이스 2 | Vercel에서 Deployment Failed가 떠요 | Ch5 |

## 케이스 1 — Claude가 엉뚱한 파일을 새로 만들어요

**증상**

`index.html` 아래에 카드 섹션 추가를 요청했는데, 응답을 보니 `index2.html`이나 `projects.html` 같은 새 파일이 폴더에 생겨 있어요. 기존 파일은 그대로고, 원하는 내용은 새 파일에 들어가 있는 상황이에요. Ch2에서 프로젝트 카드를 추가할 때 가장 많이 나오는 케이스예요.

**원인**

프롬프트가 "추가해 주세요" 수준으로 짧으면 Claude가 기존 파일 편집과 새 파일 생성 중 어느 쪽이 맞는지 스스로 판단해요. 컨텍스트가 충분하지 않으면 새 파일로 분리하는 쪽을 선택하는 경우가 있어요. 어느 파일에 넣어야 하는지 명시적으로 알려주지 않으면 생기는 문제예요. 기존 파일 이름과 위치를 프롬프트에 한 줄 명시하는 것만으로 대부분 예방할 수 있어요.

같은 문제가 반복된다면, 매 대화 첫 줄에 "파일 이름은 index.html이에요"라고 고정 문구를 넣는 것도 좋아요. 습관이 되면 새 파일 생성 실수가 거의 사라져요.

**해결 프롬프트**

파일 이름과 위치를 명시하고, 기존 구조를 건드리지 말라는 지시를 함께 넣어요. 아래 형식이 잘 작동해요.

```markdown
지금 my-portfolio/index.html 파일이 있어요.
새 파일 만들지 말고, 이 index.html 안에 "프로젝트" 섹션을 추가해 주세요.
명함 div는 그대로 두고, 명함 아래에만 append 형태로 섹션을 붙여주세요.
```

**확인**

응답 메시지에서 "Modified file: index.html" 또는 "Edited file: my-portfolio/index.html"을 확인해요. 폴더에 새 파일이 없는지도 체크하세요. 파일 탐색기(맥은 Finder, 윈도우는 탐색기)에서 `my-portfolio` 폴더를 열어 `index.html` 하나만 있는지 확인하는 것도 좋아요. 브라우저에서 `index.html`을 새로고침해서 프로젝트 섹션이 실제로 추가됐는지도 눈으로 확인해요. 혹시 새 파일이 생겼다면 아래처럼 추가 요청할 수 있어요.

```markdown
방금 만든 projects.html 파일을 지우고, 그 내용을 index.html로 합쳐주세요.
```

> **TIP:** 프롬프트 앞에 "파일 이름은 index.html이에요. 이 파일에만 추가해요" 한 줄 달면 실수가 크게 줄어요. 파일 이름과 위치를 명시하는 습관이 가장 빠른 예방책이에요.

## 케이스 2 — Vercel Import 실패 ("Deployment Failed")

**증상**

Vercel에서 GitHub 레포를 Import한 뒤 30~60초 빌드 대기가 끝나고, 초록색 대신 빨간 **Deployment Failed** 메시지가 떠요. 옆에 에러 로그 링크가 함께 표시돼요.

<!-- IMAGE: screenshot-app-a-vercel-failed.png
Vercel 대시보드에서 Deployment Failed 상태가 뜬 화면.
'View Logs' 버튼을 클릭해 ERROR 라인이 표시된 로그 패널이 열려 있는 모습.
-->

**원인**

크게 세 가지로 나뉘어요. 에러 로그 없이 원인을 추측하기 어려우니, 로그를 먼저 확인하는 게 핵심이에요.

- **(a) Framework Preset 오선택** — Import 설정 화면에서 Next.js, Create React App, Vue 등 빌드 도구가 있는 프레임워크를 고르면 Vercel이 빌드 스크립트를 실행하려 해요. 순수 HTML 파일에는 빌드 스크립트가 없으니 바로 실패해요. 로그에 `npm run build` 또는 `Build command exited with 1`이 보이면 이 경우예요.
- **(b) 파일명에 한글 포함** — `프로젝트.html`, `소개.css` 같은 한글 파일명은 서버 환경에서 인코딩 문제를 일으킬 수 있어요. 파일명은 영문 소문자와 하이픈(`-`)만 쓰는 게 안전해요.
- **(c) 파일명 대소문자 충돌** — `Index.html`(대문자 I)과 `index.html`(소문자 i)을 macOS 로컬에서는 같은 파일로 봐요. 하지만 Vercel 서버는 Linux라 엄격하게 구분해요. Vercel이 `index.html`을 찾는데 실제 파일은 `Index.html`이면 404로 실패해요.

**해결 프롬프트**

에러 로그를 Claude에게 그대로 전달하는 게 가장 빠른 방법이에요. 원인을 추측해서 설명하려 하지 말고, 로그 원문을 통째로 붙여넣는 게 훨씬 정확해요.

```markdown
Vercel에서 Deployment Failed가 떴어요. 에러 로그는 이래요:
[에러 로그 텍스트 복사 붙여넣기]

원인을 찾고, 필요한 파일 이름 변경이나 설정 수정을 해주세요.
그리고 변경 사항을 GitHub에 다시 푸시해주세요.
```

**확인**

Vercel 대시보드 Deployments 탭에서 새 Deployment 항목이 **Ready (초록색)** 상태로 뜨는지 확인해요. 보통 수정 후 다시 GitHub에 푸시하면 Vercel이 자동으로 새 배포를 시작해요. 자동 배포가 시작되지 않으면 대시보드에서 "Redeploy" 버튼을 눌러 수동으로 재배포할 수 있어요. URL을 클릭해서 브라우저에 페이지가 정상적으로 열리면 완료예요. 첫 페이지 로딩에 3~5초가 걸리는 건 정상이에요(Vercel 무료 플랜의 콜드 스타트).

> **WARNING:** 로그 원문을 복사해서 Claude에게 주는 게 스크린샷보다 훨씬 정확해요. Vercel 대시보드 → Deployments → 실패한 항목 클릭 → 'View Logs' → 붉은 ERROR 라인 전체를 복사해서 붙여넣어요. 스크린샷은 텍스트 인식이 불완전해서 오진이 나올 수 있어요.

> **TIP:** Framework Preset은 순수 HTML이면 **Other** 하나만 맞아요. Import 설정 화면에서 자동으로 프레임워크가 감지됐다면 직접 Other로 바꿔야 해요. 다른 설정은 전부 빌드 스크립트를 기대해서 실패해요.

> **NOTE:** 케이스 2는 배포 환경(Vercel)에서만 나오는 문제예요. 로컬에서 `index.html`을 더블클릭해서 여는 것과 Vercel 배포는 서버 환경이 달라요. 로컬에서 잘 보이던 게 배포 후 안 보인다면, 대소문자 불일치나 Framework Preset 문제를 먼저 의심해요.

## 케이스 3 — 이미지가 안 보여요

**증상**

브라우저에서 `me.jpg`가 들어가야 할 자리에 깨진 아이콘이 뜨거나 빈 공간으로 남아요. 로컬에서 분명히 이미지 파일을 폴더에 넣었는데도 반영이 안 되는 경우예요. Ch1에서 프로필 사진을 추가할 때 처음 마주치는 케이스이고, 배포 후에도 같은 이유로 다시 나타날 수 있어요.

<!-- IMAGE: screenshot-app-a-image-path-error.png
브라우저에서 이미지 영역에 깨진 아이콘이 뜨고,
DevTools Network 탭에 me.jpg가 404로 표시된 모습.
-->

**원인**

세 가지 패턴이 자주 나와요.

- **(a) 파일명 대소문자 불일치** — HTML에 `me.jpg`라고 적었는데 실제 파일명이 `Me.JPG`인 경우. macOS 로컬에서는 보이지만 Vercel 배포 후엔 404가 나요.
- **(b) 이미지가 폴더 밖에 있음** — `my-portfolio` 폴더 안에 `index.html`이 있는데 이미지는 바탕화면이나 다른 폴더에 있어요.
- **(c) 절대경로 사용** — HTML에 `/me.jpg`로 적으면 '서버 루트의 me.jpg'를 뜻해요. 로컬에서 파일을 더블클릭으로 열면 서버 루트가 없어서 이미지를 찾지 못해요.

**해결 프롬프트**

이미지 파일이 어느 폴더에 있는지 위치를 명확히 알려주는 게 핵심이에요. "이미지가 안 보여요"만 말하면 Claude가 경로 대신 다른 원인을 추측할 수 있어요.

```markdown
index.html에서 me.jpg가 안 보여요.
파일은 my-portfolio 폴더 안에 있어요. 이미지 경로를 상대경로 "me.jpg"로 고치고,
파일명 대소문자도 실제 파일과 맞게 수정해주세요.
```

**확인**

브라우저를 새로고침해요. 그래도 안 보이면 DevTools(F12) → Network 탭에서 `me.jpg` 요청이 **200 OK**인지 확인해요. 404가 뜨면 여전히 경로 문제예요. 200이어도 안 보이면 CSS에서 `display: none`이 적용된 게 아닌지 Elements 탭에서 확인해요.

로컬에서 더블클릭으로 열 때는 잘 보이는데 Vercel 배포 후에만 깨진다면 대소문자 불일치(원인 a)가 가장 유력해요. GitHub에 올라간 파일명과 HTML 내 경로를 한 글자씩 비교해보세요.

> **NOTE:** HTML에서 상대경로(`me.jpg`)가 기본이에요. `./me.jpg`도 같은 의미예요. `/me.jpg`는 절대경로라 '서버 루트'를 뜻하기 때문에, 로컬에서 파일을 바로 열 때는 작동하지 않아요. Vercel 배포 후에도 루트 경로 설정이 없으면 실패할 수 있어요.

> **TIP:** 이미지 파일명은 영문 소문자와 하이픈만 쓰는 게 가장 안전해요. `my-photo.jpg`처럼 단순하게 짓고, HTML 경로와 실제 파일명을 항상 같게 맞춰두세요. 대소문자 불일치를 예방하는 가장 확실한 방법이에요.

## 케이스 4 — 폰트가 이상해요 / 한글이 깨져요

**증상**

Google Fonts를 연결했는데 기본 폰트(Times New Roman이나 굴림)로 그대로 보여요. 또는 한글 텍스트가 □□□로 깨져서 나와요. 로컬에서는 멀쩡한데 Vercel 배포 후에만 이상한 경우도 있어요. Ch3에서 폰트를 바꿀 때, 그리고 Ch1에서 처음 한글 이름을 넣을 때 자주 나오는 케이스예요.

**원인**

- **(a) `<meta charset="UTF-8">` 누락** — HTML 파일에 인코딩 선언이 없으면 브라우저가 한글을 엉뚱한 방식으로 해석해요. 한글 깨짐의 가장 흔한 원인이에요. `<head>` 맨 첫 줄에 있어야 효과가 확실해요.
- **(b) Google Fonts 링크 URL 오타** — `href` 주소가 한 글자라도 틀리면 폰트를 불러오지 못해요. Claude가 생성한 URL도 가끔 오타가 나요. Google Fonts 사이트에서 직접 복사한 코드가 가장 안전해요.
- **(c) CSS `font-family` fallback 누락** — Google Fonts가 로딩되지 않았을 때 대체 폰트를 지정하지 않으면 시스템 기본 폰트가 나와요. 인터넷이 느린 환경에서도 같은 증상이 나타나요.

**해결 프롬프트**

증상이 두 가지(깨짐 + 폰트 미적용)이더라도 한 번에 요청할 수 있어요. 아래처럼 원하는 폰트 이름과 fallback 순서를 함께 알려주면 Claude가 한 번에 처리해줘요.

```markdown
index.html에서 한글이 깨지거나 폰트가 적용되지 않아요.
<head>에 <meta charset="UTF-8">이 있는지 확인하고,
Google Fonts Noto Sans KR을 최상단에 preconnect + stylesheet link로 정확히 넣어주세요.
font-family는 "Noto Sans KR", -apple-system, sans-serif 순서로 설정해 주세요.
```

**확인**

DevTools → Elements 탭에서 `<head>` 안을 펼쳐 `<meta charset="UTF-8">`와 Google Fonts `<link>` 두 태그가 모두 있는지 확인해요. 페이지를 새로 불러올 때 폰트가 0.3~1초 사이에 바뀌는 순간이 보이면 정상이에요(폰트 로딩 지연). 완전히 바뀌지 않는다면 DevTools → Network 탭 필터를 'Font'로 설정하고 폰트 파일이 200 OK로 내려오는지 확인해요.

한글이 여전히 깨진다면 파일을 저장할 때 인코딩이 UTF-8로 설정됐는지도 확인해야 해요. 메모장이나 메모 앱이 아닌 VS Code에서 저장하면 기본값이 UTF-8이라 안전해요. Claude에게 "파일을 UTF-8로 다시 저장해줘"라고 요청하는 것도 방법이에요. 윈도우 메모장은 기본 인코딩이 ANSI인 경우가 있어서 한글이 깨지는 원인이 되기도 해요. "다른 이름으로 저장" → 인코딩을 "UTF-8"로 바꿔서 저장하거나, 처음부터 VS Code를 쓰는 게 편해요.

> **NOTE:** Noto Sans KR은 한글을 포함한 폰트라 용량이 커요. `display=swap` 파라미터를 링크 URL에 붙이면 폰트 로딩 전에도 텍스트가 먼저 보여요. Claude에게 "Noto Sans KR을 font-display=swap으로 넣어주세요"라고 하면 돼요.

## 케이스 5 — 맥/윈도우에서 경로 구분자가 달라요

**증상**

Claude가 내놓은 설명에 `my-portfolio\\index.html`(백슬래시, 윈도우 방식) 또는 `/Users/yourname/Desktop/index.html`(맥 절대경로)이 섞여 있어요. 터미널 명령어도 맥용과 윈도우용이 뒤섞여 어느 걸 따라야 할지 혼란스러운 상황이에요. Ch0에서 폴더를 만들거나 터미널로 이동할 때 처음 겪는 경우가 많아요.

**원인**

Claude는 대화 시작 시점에 독자의 OS를 알지 못해요. 컨텍스트에서 OS 힌트를 찾지 못하면 어느 한쪽 스타일로 일관되게 응답하거나, 둘을 섞어 쓰기도 해요. 같은 세션이라도 새 대화를 시작하면 이전에 알려준 정보를 다시 알려줘야 해요. OS뿐 아니라 사용 중인 터미널(맥 Terminal, 윈도우 명령 프롬프트, PowerShell)도 함께 알려주면 더 정확한 안내를 받을 수 있어요.

**해결 프롬프트**

OS 정보를 대화 첫 줄에 넣어요. 길게 설명할 필요 없이 한 줄이면 돼요.

```markdown
저는 [윈도우 / 맥]을 쓰고 있어요.
제 OS에 맞는 경로 표기로 설명해주세요. 터미널 명령어도 제 OS 기준으로요.
```

**확인**

이후 응답에서 경로 구분자가 일관되게 나오는지 확인해요. 맥이면 `/`(슬래시), 윈도우면 `\`(백슬래시)여야 해요. 현재 폴더 위치는 터미널에서 맥은 `pwd`, 윈도우(명령 프롬프트)는 `cd`만 입력하고 엔터 치면 확인할 수 있어요. 안내대로 따라했는데 "그런 경로가 없다"는 오류가 난다면 OS 구분자를 다시 확인해 보세요. 경로 문제의 절반은 구분자 혼용이에요.

> **TIP:** 첫 대화에서 한 번 OS를 알려주면 같은 세션 동안 Claude가 기억해요. "저는 맥북 써요" 한 줄이면 충분해요. 새 세션(터미널을 닫고 다시 `claude`를 실행)이면 다시 한 줄 알려주는 게 편해요.

## 케이스 6 — Claude의 답이 중간에 끊겨요

**증상**

긴 CSS나 HTML 응답이 `...`으로 끝나거나 갑자기 멈춰요. 코드 블록이 `</style>`나 `</body>` 없이 닫히지 않은 채로 끝나요. 파일에 반만 붙여넣어서 레이아웃이 깨지기도 해요. Ch3에서 스타일을 한꺼번에 많이 바꾸거나, Ch4에서 반응형과 다크모드를 동시에 요청할 때 나타나는 경우가 많아요.

**원인**

한 번의 응답으로 출력할 수 있는 내용에 한도가 있어요. 파일 전체를 다시 써달라거나 여러 변경을 한꺼번에 요청하면 한도에 근접해서 중간에 끊길 수 있어요. 특히 CSS가 길거나 HTML 구조 전체를 다시 요청하는 경우에 자주 나와요. 대화가 이미 길어진 상태(앞서 주고받은 메시지가 많을 때)에도 응답 한도에 더 빨리 도달할 수 있어요.

**해결 프롬프트**

끊긴 지점부터 이어달라고 요청해요. 전체를 다시 요청하면 같은 자리에서 또 끊릴 수 있어요.

```markdown
방금 답이 중간에 끊겼어요.
끊긴 지점 이후부터 이어서 써주세요. 전체를 다시 쓰지 말고, 나머지만요.
```

또는 처음부터 요청 범위를 좁혀서 다시 시작해요. 한 번에 하나씩 바꾸는 게 끊김을 예방하는 가장 확실한 방법이에요.

```markdown
색만 먼저 바꿔주세요. 폰트는 다음 대화에서 할게요.
```

**확인**

파일에 코드 블록이 정상적으로 닫혔는지 확인해요. `</style>`, `</body>`, `</html>` 태그가 모두 있어야 해요. 브라우저에서 새로고침한 뒤 레이아웃이 깨지지 않으면 완료예요. DevTools → Elements 탭에서 HTML 트리가 정상적으로 펼쳐지는지 확인하는 것도 좋아요.

대화가 너무 길어졌다면 새 세션을 시작하는 것도 방법이에요. 새 세션에서 "지금 index.html 파일이 이렇게 생겼어요"라며 현재 파일 내용을 붙여넣고, 원하는 변경만 좁게 요청하면 훨씬 깔끔하게 완성돼요. 긴 대화보다 짧고 명확한 대화 여러 번이 결과물 품질도 더 좋은 경우가 많아요.

> **WARNING:** 답이 끊긴 상태에서 그냥 파일에 붙여넣고 저장하면 HTML 구조가 깨져요. 태그 쌍이 맞지 않아 브라우저가 레이아웃을 엉뚱하게 그릴 수 있어요. 꼭 이어쓰기 요청으로 코드를 완성한 뒤에 파일에 적용해요.

> **TIP:** 끊김 예방을 위해 처음부터 요청을 작게 쪼개는 습관이 도움돼요. "색·폰트·여백을 한 번에 바꿔주세요" 대신 "배경색만 먼저 바꿔주세요"처럼 한 가지씩 요청하면 끊킬 확률이 크게 낮아지고, 결과물도 더 예측 가능해요.

## 자주 묻는 질문 (FAQ)

**Q. 이력서에 넣을 Vercel URL을 나중에 바꾸면 어떻게 되나요?**

Vercel 대시보드 → Settings → Domains에서 기본 제공 URL(`yourname.vercel.app`)은 유지하면서 추가 커스텀 주소를 붙이거나, 프로젝트명을 변경해서 URL을 바꿀 수 있어요. 단, 이미 이력서에 넣은 링크는 제출 후엔 수정이 안 되니, URL을 바꿨다면 새로 지원하는 곳부터 새 URL을 쓰면 돼요. 가장 안전한 방법은 처음 배포할 때 URL을 확정하고 이력서에 넣는 거예요.

**Q. `my-portfolio` 폴더를 다른 이름으로 바꿔도 되나요?**

로컬 폴더 이름은 Vercel 배포 URL과 직접 연관이 없어요. Vercel URL은 GitHub 레포 이름과 Vercel 프로젝트명을 기준으로 정해져요. 로컬 폴더명을 바꿔도 이미 배포된 사이트에는 영향이 없어요. 단, Claude에게 "my-portfolio 폴더에 있어요"라고 말했다면 이름이 바뀌었을 때 다시 알려줘야 혼선이 없어요.

**Q. 핸드폰에서만 다크모드가 이상해요.**

시스템 다크모드(`prefers-color-scheme: dark`)와 Ch4에서 만든 버튼 토글이 충돌하는 경우예요. `prefers-color-scheme` 미디어쿼리를 명시적으로 세팅해서 충돌을 해소할 수 있어요. Claude에게 "핸드폰 시스템 다크모드와 내 토글 버튼이 겹쳐요. prefers-color-scheme 미디어쿼리로 동기화해주세요"라고 요청해 보세요.

**Q. Vercel이 GitHub 레포를 목록에서 못 찾아요.**

Vercel Import 화면에서 레포가 보이지 않으면 GitHub App 권한 문제예요. 화면 하단의 "Adjust GitHub App Permissions" 링크를 클릭하면 GitHub 권한 설정 페이지로 이동해요. 거기서 해당 레포를 Vercel이 접근할 수 있도록 허용해 주면 돼요. 그래도 안 된다면 Vercel에서 로그아웃 후 다시 GitHub로 로그인해 보세요.

**Q. Claude가 내 파일을 읽지 못하고 "파일이 없다"고 해요.**

Claude는 현재 작업 폴더를 기준으로 파일을 찾아요. `claude`를 실행한 폴더가 `my-portfolio`가 아니면 파일을 찾지 못해요. 터미널에서 `my-portfolio` 폴더로 이동한 뒤 다시 `claude`를 실행해 보세요. 맥은 `pwd`, 윈도우는 `cd`로 현재 위치를 확인할 수 있어요. 또는 Claude에게 "지금 내 파일 구조를 알려줄게요"라며 파일 내용을 직접 붙여넣는 방법도 있어요. 파일 경로보다 내용을 직접 주는 게 오히려 더 빠를 때도 많아요.

**Q. 포트폴리오를 여러 페이지로 나누고 싶어요.**

이 책의 실습은 `index.html` 하나짜리 싱글 페이지 구조예요. 여러 페이지로 나누는 건 가능하지만 링크 관리와 네비게이션이 복잡해져요. 취준생 포트폴리오라면 싱글 페이지가 훨씬 유리해요. 스크롤 한 번에 이름, 프로젝트, 연락처까지 보여주는 게 면접관 입장에서 편하고, 배포 후 관리도 단순해요. 포트폴리오를 완성한 뒤 더 도전하고 싶다면 부록 C에 확장 아이디어가 있어요.

## 에러 메시지를 효과적으로 전달하는 법

에러를 Claude에게 가져갈 때 세 가지만 함께 알려주면 해결 속도가 달라져요. 첫째, **어디서 에러가 났는지** (브라우저 화면인지, Vercel 대시보드인지, 터미널인지). 둘째, **에러 메시지 원문** (빨간 글씨 전체, 스크린샷보다 텍스트 복사가 정확해요). 셋째, **직전에 한 작업** ("폰트를 바꾸려다 이렇게 됐어요"). 이 세 가지를 갖추면 Claude가 거의 한 번에 원인을 찾아줘요.

## 추가 도움이 필요하다면

여기서 못 찾은 에러라면 Claude에게 에러 메시지 전문을 그대로 보여주는 게 가장 빠른 해결이에요. 스크린샷과 함께 "이 에러 원인 찾고 고쳐줘" 한 줄이면 충분해요. 영어 에러 메시지도 괜찮아요. Claude가 번역과 해석을 같이 해줘요. 에러 내용을 설명하려고 애쓰지 않아도 돼요. 그냥 복사해서 붙여넣는 게 가장 정확해요.

바이브 코딩의 매력 중 하나가 바로 이거예요. 에러 메시지도 Claude가 읽어줘요. 처음 보는 영어 로그라도 한 줄씩 번역하고 어느 파일의 몇 번째 줄에서 문제가 생겼는지 짚어줘요. 무서운 빨간 글씨도 Claude에겐 평범한 단서예요. 에러가 나온 상황을 짧게 설명하고("이미지를 추가하려다 이 에러가 났어요"), 에러 원문을 그대로 붙이는 게 황금 조합이에요.

그래도 해결이 안 되는 경우엔 새 대화를 열고 처음부터 다시 설명해보는 것도 방법이에요. 대화가 길어질수록 Claude가 앞의 내용을 압축해서 처리하기 때문에, 새 세션에서 깔끔하게 시작하면 더 정확한 답을 받는 경우가 많아요. "내 포트폴리오 파일이 이렇게 생겼어요. 이 에러가 나요"처럼 현재 파일 상태와 에러를 함께 알려주면 돼요.

막힌다고 멈추지 않아도 돼요. 어떤 에러가 나왔는지 그대로 Claude에게 가져가면, 대부분의 문제는 대화 한두 번 안에 풀려요. 포트폴리오를 완성한 사람들도 처음엔 다 같은 에러를 겪었어요. 막혔다는 건 그만큼 가까이 왔다는 뜻이에요.

여기까지 읽었다면 포트폴리오가 거의 다 된 거예요. 이제 URL을 이력서에 붙여넣을 시간이에요.
