# 부록 C: 더 알아보기

이 책을 여기까지 따라왔다면 지금 손에 쥔 게 꽤 돼요.

이름·사진·한줄소개가 담긴 명함 페이지, 프로젝트 카드 3장 그리드, 내 취향으로 고른 색·폰트·여백, 모바일에서도 잘 보이는 반응형 레이아웃, 불 끄면 바뀌는 다크모드 토글, 그리고 이력서에 붙일 수 있는 공개 URL 한 개. 이 모두가 대화만으로 만들어진 거예요.

여기서 멈춰도 충분해요. 지금 상태의 포트폴리오는 실제로 쓰이는 도구예요. 링크를 걸어두고, 이력서에 적고, 면접관에게 보여줄 수 있는 결과물이에요.

그래도 더 가고 싶다면, 아래 다섯 갈래의 길이 있어요. 억지로 다 갈 필요는 없어요. 지금 내 상황에서 가장 당기는 방향 하나만 먼저 골라보세요.

---

## 1. "자소서도 Claude로 써볼까?"

이 시리즈에는 **Vibe Coding** 방식으로 자소서를 쓰는 책이 따로 있어요. 지금 포트폴리오 공개 URL이 생겼으니, 자소서에 그 링크를 넣는 게 자연스러운 다음 스텝이에요. "제 포트폴리오는 여기서 확인하실 수 있습니다"라는 한 줄을 자소서 안에 박아두는 거예요.

자소서 책도 같은 Claude Code 환경에서 만들어요. 새 도구를 배울 필요가 없어요.

이 책에서 익힌 대화 방식 — 취향을 구체적으로 전달하기, 한 번에 하나씩 요청하기, 결과물을 보고 방향을 조금씩 틀기 — 이 감각이 자소서 쪽에서도 그대로 작동해요. "나는 이런 사람이에요"를 Claude에게 설명하는 방식이, 포트폴리오 감성을 잡을 때와 크게 다르지 않거든요.

> **TIP:** 자소서 책을 이미 읽고 나서 이 포트폴리오 책을 보고 계신 분이라면, 두 책의 접근법이 뿌리가 같다는 게 체감이 될 거예요. 취향을 대화로 전달하는 것, 한 번에 완성하려 하지 않는 것 — 이게 **Vibe Coding**의 핵심 감각이에요.

## 2. 지금 포트폴리오를 더 키워보는 세 가지 방향

### 2-1. 블로그/노트 페이지 추가

카드 섹션 아래에 짧은 기록들을 달아두면 포트폴리오가 훨씬 풍성해져요. 완성된 결과물이 아니라 "요즘 이런 걸 공부하고 있어요" 수준의 짧은 노트도 충분해요. 날짜, 제목, 2~3줄 설명으로 구성된 카드 서너 개면 돼요. 기술 블로그를 따로 운영하지 않아도, 포트폴리오 안에 간단하게 붙여두는 것만으로 "계속 뭔가를 하는 사람"이라는 인상을 줄 수 있어요.

```text
index.html 맨 아래에 최근 기록을 2~3개 보여주는 간단한 노트 섹션을 추가해줘.
각 항목은 날짜, 제목, 2~3줄 짧은 설명으로 구성해줘.
지금 있는 카드 섹션의 톤과 폰트를 그대로 유지하고,
기존 레이아웃 구조는 건드리지 말고 그냥 맨 아래에 이어 붙여줘.
```

### 2-2. 연락처/문의 섹션

포트폴리오를 본 사람이 연락하고 싶을 때 어디로 가야 할지 모르면 기회를 놓쳐요. 이메일 링크 하나와 GitHub/LinkedIn 텍스트 버튼 두 개면 충분해요. 아이콘 이미지 파일을 따로 추가하는 것보다 텍스트 버튼이 훨씬 단순하고, 접근성 측면에서도 더 좋아요.

```text
index.html 맨 아래에 연락처 섹션을 추가해줘.
내 이메일은 [내이메일]이고, GitHub 주소는 [내GitHub주소]야.
텍스트 버튼 형태로 만들어줘 — 아이콘 없이, 글자만.
지금 페이지 톤에 어울리는 간결한 스타일로 해줘.
```

### 2-3. 방문자 통계 — 아주 가볍게

Vercel 대시보드에는 기본 Analytics 탭이 있어요. 배포된 프로젝트 페이지에서 "Analytics" 메뉴를 열면 방문자 수와 접속 경로를 대략 볼 수 있어요. 코드를 따로 심거나 설정 파일을 건드릴 필요가 없어요. 이력서에 링크를 넣고 지인에게 공유하기 시작했을 때, "실제로 얼마나 보는지" 가볍게 확인하는 용도로만 쓰면 충분해요.

> **NOTE:** Analytics 탭의 세부 기능은 Vercel 플랜에 따라 다를 수 있어요. 무료 플랜에서도 기본 방문자 수는 확인할 수 있어요.

## 3. vercel 명령어 한 줄 배포

Ch5에서 우리는 Vercel 웹사이트의 Import 버튼을 눌러 배포했어요. 브라우저에서 GitHub 레포를 연결하고, Framework Preset을 Other로 고르고, Deploy를 누르는 흐름이었어요. 이게 가장 눈으로 확인하기 쉬운 방법이에요.

터미널에서 `vercel` 명령어 한 줄로 배포하는 길도 있어요. 이 책에서는 다루지 않았지만, 터미널에 익숙해질수록 자연스럽게 궁금해지는 방향이에요. "vercel CLI"라고 검색하면 공식 문서에서 사용법을 찾을 수 있어요.

다만 굳이 지금 배울 필요는 없어요.

> **NOTE:** CLI가 반드시 더 빠르거나 좋은 건 아니에요. 지금처럼 `index.html`을 고치고 GitHub에 푸시하기만 하면, Vercel이 자동으로 다시 배포해줘요. 코드 한 줄 바꿀 때마다 Claude에게 푸시를 맡기면 되니까, 이게 여전히 가장 단순한 흐름이에요.

## 4. Tailwind CDN이라는 또 다른 선택지

Ch3에서 색·폰트·여백을 바꿀 때, CSS 변수와 직접 작성한 클래스들로 감성을 잡았어요. 이게 이 책의 기본 방식이고, 외부 의존 없이 내 파일 안에서 모든 게 해결되는 가장 단순한 방식이에요.

Tailwind CSS라는 스타일 도구가 있는데, `<script>` 태그 한 줄을 `index.html`에 추가하는 것만으로 간단하게 얹어볼 수 있어요. 설치 과정 없이 실험해볼 수 있다는 게 장점이에요. 지금 포트폴리오에 그대로 붙여서 동일한 룩을 Tailwind 방식으로 다시 짜보는 것도 **Vibe Coding** 감각을 키우는 좋은 연습이에요.

```text
지금 index.html에 Tailwind CDN을 얹어줘.
현재 디자인의 색·폰트·레이아웃 느낌을 최대한 유지하면서
기존 CSS 클래스를 Tailwind 유틸리티 클래스로 바꿔줘.
내부 <style> 태그는 최소화해줘.
```

> **WARNING:** CDN 방식은 실험과 학습용으로는 충분해요. 다만 실서비스 수준의 최적화가 필요해지면 빌드 단계가 필요해지고, 그때부터는 다른 책이 필요해요. 지금은 "이런 게 있구나" 정도로만 알고 넘어가도 돼요.

## 5. 내 이름으로 된 주소를 쓰고 싶다면

`내이름.vercel.app` 주소로도 충분하지만, `내이름.com` 같은 도메인을 연결하면 조금 더 프로페셔널한 인상을 줄 수 있어요.

방향만 간단히 짚어볼게요. 도메인은 Namecheap, Cloudflare 같은 서비스에서 연 단위로 구매해요. 구매한 뒤, Vercel 프로젝트의 "Domains" 메뉴에서 도메인을 추가하면 Vercel이 필요한 DNS 레코드 값을 알려줘요. 그 값을 도메인 서비스 쪽 DNS 설정에 입력하면 연결이 완료돼요. 자세한 절차는 Vercel 공식 문서의 "Custom Domains" 항목에 단계별로 나와 있어요.

> **TIP:** 지금 당장 필요하지 않으면 나중으로 충분히 미뤄도 돼요. 도메인은 연 단위 결제라 포트폴리오가 어느 정도 자리를 잡은 뒤에 결정해도 전혀 늦지 않아요. `vercel.app` 주소도 이력서에 넣기에 충분해요.

## 마무리

환경을 세팅하고, 명함을 만들고, 카드를 얹고, 감성을 입히고, 반응형과 다크모드를 손에 넣고, 공개 URL까지 가져왔어요. 대화만으로요.

처음 터미널 앞에 앉았던 순간과 지금을 비교해보면, 그 사이에 꽤 많은 게 달라졌어요. 코딩을 배운 게 아니라, 대화하는 법을 배웠어요. 그리고 그 대화로 실제로 존재하는 무언가를 만들어냈어요.

이게 **Vibe Coding**이에요.
